#### 1. eex 文件头与 场景 索引

`cczEditor2Doc.cpp::OnSaveDocument`

- 保存时固定先写入 10 字节头：`45 45 58 00 01 02 00 00 00 00`。
- 紧接着为每个 场景 预留 4 字节偏移表。
- 每个 场景 实际起点会回填到 `10 + sceneIndex * 4`。
- 每个 场景 内部先写 2 字节 `section_cnt`，然后顺序写每个 段落。

结论：

- 当前工具把 `.eex` 视为 `EEX\\0 + Scene offset table + Scene body` 是对的。
- `HeaderSize` 不能再只当“候选”；旧工具的保存逻辑明确以 `10 + sceneCount * 4` 形成头部。

#### 2. 段落 是带长度前缀的嵌套块

`cczEditor2Doc.cpp::OnSaveDocument`

- 段落 开头先写 2 字节总长度预留。
- 段落 内命令顺序写入。
- 一旦某条命令带子节点，就继续进入子节点并额外写一个 2 字节长度预留。
- 当子块结束时，回填 `当前块长度 = endPos - blockStart - 2`。

结论：

- 段落 本身不是简单命令数组，而是“长度前缀块 + 可递归子块”。
- 旧工具里的子事件嵌套，底层依赖的是“命令后可接一个子块长度+子命令流”。
- 当前工具仍不能把探针结果直接当可写命令树。

#### 3. 旧工具读取时的事件/子事件建树规则

`cczEditor2View.cpp::CreateFileTree`

已确认：

- `head=true` 时处于 段落 头部。
- 遇到 `id == 0` 且 `head == true`：
  - 头部结束；
  - 当前父节点切到这个 `00` 命令之下；
  - 额外跳过 2 字节长度。
- 遇到 `id == 1`：
  - 仅设置 `zsj_flag = true`，表示“刚看到子事件设定”。
- 下一条命令若 `code_test[id] != 0` 且 `zsj_flag == true`：
  - `zsj_sum++`；
  - 当前父节点切到“这条命令”之下；
  - 额外跳过 2 字节长度。
- 遇到 `id == 0` 且 `zsj_sum > 0`：
  - 当前子事件块结束；
  - 父节点回到上一层。

关键结论：

- `01` 本身只是“子事件设定”标志，不是直接承载子树的节点。
- 真正挂子树的是 `01` 后面那条满足 `code_test[id] != 0` 的命令。
- `00` 有两种角色：
  - 段落 头部结束后，作为“正文根”并带一个子块长度；
  - 子事件块结束标记。

因此，当前工具的数据模型应保留下面语义，但可视界面不应额外制造这些中文包装节点：

- 旧源码实际只插入 `Scene`、`Section` 和每条真实指令节点。
- `00` 作为 段落 头部结束时，源码把后续父节点切到这个 `00` 指令下，而不是再创建“正文根/事件正文”节点。
- `01` 作为“子事件设定”时只设置标志；它后面满足 `code_test[id] != 0` 的命令才承接子块，源码直接把父节点切到该命令下，不创建“子事件主体/载体”包装节点。
- 界面对齐旧版时应显示为 `文件路径 -> Scene -> Section -> 指令`，子事件通过真实指令节点嵌套；内部语义写入 tooltip/详情或数据模型，不抢占树视图。

#### 4. 无条件跳转 76 的真实保存语义

`cczEditor2Doc.cpp::OnSaveDocument`
`cczEditor2View.cpp::CreateFileTree`

读取时：

- `code_off[ord] = 命令实际文件偏移`
- `76` 先读出相对位移 `dif`
- 目标偏移按 `pos + dif + 8` 计算
- 再在 `code_off[]` 中反查对应命令 ord，写回 `int_data[0]`

保存时：

- 保存前先记录每条命令新 ord
- `76` 保存时先记下当前位置 `jmp_pos`
- 文件写完后回填 4 字节跳转位移：
  - `targetOffset - currentOffset - 8`
- 保存完成后再按新 ord 刷新树中显示

结论：

- 旧工具里的 `76` 不是简单“事件块编号”，而是基于命令偏移/ord 的跳转。
- 当前工具把它泛化成“跳转目标候选”可以保留，但要明确这是“命令序号/偏移回填语义”，不是普通文本编号。

#### 5. `code_instruct` / `code_test` 的直接价值

`cczEditor2View.h`

- `code_instruct[124][13]`：旧工具的命令参数布局表。
- `code_test[124]`：旧工具对命令类别/可否作为子事件载体的判定表。

直接可用结论：

- 现有 C# `ScenarioCommandParameterTemplateService` 可以继续借用命令名和用途，但后续若要逼近旧工具，应该把 `code_instruct` 逐步转为结构化参数长度表。
- `code_test[id] != 0` 可作为“这条命令可承接子事件块”的已确认信号，不应再只靠 `01` 单独入栈。

##### 2026-06-05 战场部署大块的特殊长度

- `code_instruct` 每条只有 13 个槽，但旧工具对部分命令不是简单读取一遍：
  - `46 友军出场设定`：按前 11 个布局槽循环 20 组，合计 220 个参数。
  - `47 敌军出场设定`：按前 12 个布局槽循环 80 组，合计 960 个参数。
  - `4B 我军出场设定`：按 5 个布局槽读取一条记录，不是大块循环。
- 任何只读探针或候选表都必须同步这两个循环规则。否则 `46/47` 会被当作短命令截断，后续命令边界整体错位，战场编辑器会表现为“能看到出场命令名，但人物、坐标、人工智能 解析不正确”。
- 旧结构参数不是全 16 位：`04` 布局槽是 32 位整数参数。`46` 的 X/Y 分别是第 3/4 个逻辑参数，`47` 的 X/Y 分别是第 4/5 个逻辑参数，均来自 32 位槽；如果只展开 16 位词或按参数对象下标错位写入，会把出场标志误当坐标。
- 当前 `LegacyScenarioReader` 与 `ScenarioStructureProbeReader` 均已同步该规则；战场页把 `46` 拆为最多 20 条友军候选、`47` 拆为最多 80 条敌军候选、`4B` 拆为单条我军候选。`4B` 是我军出战选择排列，第 1 槽是出战顺序，不是人物号。
- 字段写回边界：已自动/手动绑定到 `Scene/Section/Command/Record` 的 `46/47` 可写人物、X/Y 和人工智能；`4B` 只写 X/Y、方向、隐藏标志，不改出战顺序/人物槽。找不到可用槽的纯拖放、等级/装备/援军等未完全确认字段仍只做候选核对和项目侧备注。

4. 说明和源码对照里明确：
   - `76` 基于命令偏移/ord 回填
   - `01` 只是子事件设定

5. 旧版完整树模式下已开放保守普通命令编辑：
   - 可从 `CczString.ini` 选择普通命令，添加到正文末尾、在普通命令前后插入、追加到已有子块。
   - 可删除无子块、未被 `76` 引用、且非结构分界的普通命令。

6. 所有节点增删和参数修改先写入内存中的 `LegacyScenarioDocument`，必须点击 `完整保存剧本` 才落盘；写入继续依赖备份、临时文件、旧版 读取器 复读校验和结构化报告。

##### 2026-06-06 `UpdateShow()` 中文摘要迁移

旧源码 `cczEditor2View.cpp::UpdateShow(HTREEITEM me)` 不是简单把参数列表拼到树节点上，而是按命令 id 分支把 `int_data` / `long_char_data` 映射成创作者可读文本。本次迁移新增 `LegacyScenarioCommandDisplayFormatter`，并让剧本、战场、R 场景三处旧版脚本树共用。

关键口径：

- 文本类命令把内部换行显示为 `\n`，直接挂在命令行后，不再创建独立“文本参数”子节点。
- `05` 变量测试按旧版 25/25 两组变量数组显示，空组显示 `无`。
- 人物选择走旧版 `Per1CodeToList` / `Per2CodeToList`，显示人物名或特殊对象，不显示内部负数编码。
- 坐标、阵营、天气、物品、兵种、状态、比较符、运算符、单挑动作等均优先使用 `LegacyMfcDialogDataSources` 从旧版 `CczString.ini` 与项目资源加载的中文映射。
- 注意旧源码里的 `id == 119/120/121` 对应当前命令字节 `77/78/79`，不能误按十进制 `77/78/79` 迁移。
- `76` 在当前 C# 读写链路仍按无条件跳转 ord/偏移回填语义处理；显示层只负责摘要，不改变保存语义。

已覆盖验证样例：

- `05 变量测试`：显示 `Var20;无`。
- `25 武将进入指定地点测试`：显示中文人物/阵营与坐标 `(13,12)`，不出现 `P0=`。
- `36 武将状态测试`：显示人物名、`HPCur` 与比较符 `=`。

- `Dialog_48::OnBnClickedOk()` 对应 `30 武将出现`：`int_data[0]=人物`，`int_data[1]=X`，`int_data[2]=Y`，`int_data[3]=方向`，`int_data[4]=动作`。
- `code_instruct[30]` 为 `2,4,4,2B,D`，因此 X/Y 是旧版 32 位数值槽，不是普通 16 位槽。

当前实现边界：

- R 场景页只把 `30 武将出现` 自动预加载为画布角色；移动、转向、动作、形象改变继续保留在候选表中供核对。
- 右键画布角色进入可编辑状态，拖拽只同步已有 `30` 命令的 X/Y 槽；纯拖放新增角色、非 `30` 命令绑定项仍只保存项目侧草稿。
- 双击画布角色定位左侧 R 剧本树中的对应真实命令，方便继续用旧式对话框 编辑。
- 坐标同步只修改内存中的 `LegacyScenarioDocument`；必须点击 `完整保存R剧本` 才按 `LegacyScenarioWriter` 写入 `.eex`，并继续执行备份、临时文件替换前复读和报告输出。

##### 2026-06-06 `IDR_POPUP_EDIT` 右键菜单迁移结论

旧源码菜单资源：`cczEditor2.rc::IDR_POPUP_EDIT`。

旧版右键菜单顺序：

- `修改(&E)\tCtrl+E`
- `添加(&I)\tCtrl+I`
- `添加子事件(&S)\tCtrl+O`
- `步进复制(&D)\tCtrl+D`
- `批量修改(&R)\tCtrl+R`
- `删除(&D)\tDelete`
- `上移(&U)\tCtrl+Up`
- `下移(&D)\tCtrl+Down`
- `剪切(&T)\tCtrl+X`
- `复制(&C)\tCtrl+C`
- `粘贴(&P)\tCtrl+V`
- `全部展开\tCtrl+Q`
- `跳转到...`

关键源码行为：

- `OnContextMenu()` 只弹 `IDR_POPUP_EDIT`，没有额外备注或参数表菜单项。
- `OnEditAdd()` 选中指令时打开 `Dialog_SelectCode`；真正插入位置由 `FindLast(cur_item)` 决定，即插入到当前指令前。
- `Dialog_SelectCode::OnBnClickedOk()` 禁止手动添加 `0/1`；若当前指令处于 段落 顶层且当前是 `2` 号指令，则提示“不允许在这里添加指令”。
- 段落 顶层新增命令需满足 `code_test[id] >= 2`，旧源码还额外允许 `72/77/78`。
- 若 `FindLast(cur_item)` 是 `1` 号子事件设定标志，则禁止在 `01` 和其后承载命令之间插入普通命令。
- `OnEditAddi()` 要求当前节点父级不能是 场景/段落；也就是子事件只能在正文或已有子事件块内创建。
- `CreateZsj(id, parent, after)` 顺序是：先创建 `01`，再创建子事件承载命令，最后在承载命令子树内创建 `00`。
- `OnCopy()` 禁止复制 `0/1`；`OnPaste()` 默认同样按 `FindLast(cur_item)` 插到当前节点前，复制对象带子树时按需补 `01`。
- `OnMoveUp/OnMoveDown()` 禁止移动 `0/1`，并禁止移动 段落 顶层头部指令；下移遇到 `00` 视为到尾。
- `OnEditDuplicate()` 不支持 `46/47/05/00/01/02`，也不支持带子树的命令。
- `OnEditBatch()` 只在旧版 checkbox 多选模式下可用；当前整合包继续暂缓批量/大规模修改。

当前 C# 对齐：

- `MainForm.ConfigureLegacyStyleScriptTreeContextMenu()` 生成旧版同序菜单。
- `Ctrl+E/I/O/D/R/X/C/V/Q`、`Delete`、`Ctrl+Up/Down` 走同一套菜单逻辑。
- `AddLegacyScriptCommandBeforeSelected()`、`AddLegacyScriptSubEventBeforeSelected()`、`PasteCopiedLegacyScriptCommandNearSelected(beforeSelected: true)` 复刻旧版“当前指令前插”的结构语义。
- `LegacyScriptCodeTestTable` 来自旧 `code_test[124]`，用于子事件承载候选和 段落 顶层新增限制。

##### 2026-06-08 剧本编辑器 v0.24 静态对照结论

`剧本编辑器v0.24` 当前只有原生 `cczEditor_v0.24.exe` 和配置文件，没有源码；因此源码级 `.eex` 读写规则仍以 `a新剧本编辑器v0.23` 源码为准，v0.24 只作为静态取证和交互习惯补充。
