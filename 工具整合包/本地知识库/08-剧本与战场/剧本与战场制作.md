迁移到当前 6.5 工具链的短结论：

- R/S 教程适合补充“怎么读剧本”和“哪里容易坏”，不替代旧源码、docx、样本和烟测。
- R/S 剧本都按 `Scene -> Section -> 指令/子事件` 阅读；当前工具的真实结构仍以 `LegacyScenarioReader` 读出的旧版完整树为准。
- S 剧本可先按“战前设置、战中处理、战后处置”三段式阅读：战前重点看 `44/45/46/47/4B`、背景/绘图、胜败条件、菜单；战中重点看触发/回合/人数/行动方/胜败测试；战后重点看菜单、调色板、`11` 跳转和 `0D` 收尾。
- R 入 S 必须结合本地样本判断：有的关卡使用出战测试后的 `11 剧本跳转`，有的关卡使用“剧情/出战设置 Scene + 玩家切换 Scene + `0D 结束Scene`”让引擎接入对应 S。不能把 `07 + 11` 自动套到所有首关。
- 子事件后必须接能展开的承接命令；这与旧源码 `01 + code_test[id] != 0` 的结构规则一致。当前界面继续保护 `00/01` 和承接命令配对。
- `即时判定`、`76 无条件跳转`、带跳转的子事件复制、只有一个 段落 的跳转循环等都列为高风险；当前仍不开放手填跳转或跨 段落 自动重排。
- 教程中的“另存后重新打开确认不报错”对应当前工具的备份、临时文件、`LegacyScenarioReader` 复读和结构化报告。

#### 剧本树防闪烁与防重复重读规则

- 剧本制作页左侧树的当前权威状态是 `_currentLegacyScriptDocument`、`_currentScriptStructure` 和 `_currentScriptTextEntries`；只要目标剧本未切换且内存模型有效，就不得为了保存、定位或刷新辅助窗口重新读取整个 `.eex`。
- 文本保存后只更新对应 `ScenarioTextEntry`、文本树节点和当前详情；不要重建整棵树。
- 单条命令参数或旧式对话框修改后优先使用命令级局部刷新；同一 Section 内的插入、删除、移动、粘贴、文本导入等结构变化优先刷新该 Section。
- 变量窗口定位、战场页跳转、独占套装跳转等跨页面入口，如果目标剧本已经是当前加载剧本，只允许定位节点，不得再次调用完整加载流程。
- 刷新剧本索引时必须保留仍存在的当前剧本、选中节点、展开状态和滚动位置；不能因为重新读取文件列表而清空当前树。
- 允许完整重读的情况仅限：用户显式切换到另一个剧本、文件被外部修改或删除、旧版完整树模型失效、局部刷新无法安全定位结构边界。

- 左侧树与旧剧本编辑器的阅读顺序进一步对齐：
  - `Scene -> Section -> 测试条件 -> 正文根 -> 事件正文 / 子事件`
  - 文本线索优先归属到 `Section文本线索`，只把无法稳定归到某个 段落 的剩余文本放到 `Scene补充文本线索` 或 `未归属文本线索`。
  - 选中 `Scene` / `Section` 节点时，右侧只显示该结构节点详情，不再让命令/文本表反向选中其它对象。
  - 选中文本节点时，仅设置当前文本参数并刷新右侧编辑框；要复制命令必须选中命令树节点本身。
- 文本线索支持直接写入当前项目 `RS\*.eex`：
  - 入口：剧本制作页 `保存选中文本`。
  - 写入方式：`ScenarioTextWriter.SaveInPlace`。
  - 写入限制：只支持 GBK 字节数不超过原容量的原偏移短写回。
  - 保存前自动备份目标 eex。
  - 保存后重新读取同偏移校验。
  - 结构化报告类型：`R/S eex 剧本文本写回`。
- 旧高级文本探针页的批量保存按当前项目可写口径执行，但仍受原 GBK 容量、目标路径、备份、复读和报告护栏限制；测试副本用于隔离验证和发布前对比，不再作为所有写入的前置条件。
- 提示：剧本制作页的“剧本列表”在界面上按传统流程排序为 `R_00..R_nn` 在前、`S_00..S_nn` 在后，同类内按编号排序，避免按字典序出现 `R_1`/`R_10`/`R_2` 的错觉。

#### 6.5 R_00 原版容器与独立基底纯净首关规则

- 原版 `R_00.eex` 复读为 `5 Scene / 7 Section / 8528 Commands / 116206 bytes`，承担开局初始化、历史/指南、模式选择和正式剧情入口。这是原版样本事实，不等于独立精简基底必须保留全部启动骨架。
- 2026-06-19 《三英战龙帝》实机复盘：多 Scene 或保留原版 `5 Scene / 7 Section` 启动骨架会残留启动 / 菜单 / 配置分支，污染独立首关执行流；把文学分幕写成多个真实 Section 时，实机会从非第一 Section 入口开始执行，出现“直接从最后一幕入阵”的问题。

#### 2026-06-01 剧本制作页 场景 树 界面：按“事件/子事件结构”优化（启发式）

背景知识（来自旧剧本编辑器语法说明）：

- 一个剧本可由若干部分组成，每一部分内部包含多个事件；每个事件分为“测试条件（可多个）”与“事件体（可多个，可包含子事件）”，并由事件结束标志分隔。
- 子事件以“子事件开始标志”开头，测试条件只能一个，子事件内部也可继续嵌套子事件，最后以事件结束标志结束。

当前 CCZModStudio 的实现策略：

- `ScenarioStructureProbeReader` 仍是只读探针：只可靠识别 `0D` 结束 场景、`0C` 结束 段落，并把 `00` 标为“事件结束/分隔标记候选”。参数长度未完全确认，不作为写回依据。
- 剧本制作页左侧树在界面层新增 “事件块” 分组：
  - `Scene -> Section -> 事件 -> (测试条件 / 事件 / 分隔标记候选) -> Command`
  - 事件分割规则（启发式）：在同一个 段落 下，以 `0000` 命令作为事件结束/分隔候选，把命令切成多个“事件”。
  - “测试条件”块规则（启发式）：事件起点到第一个 `0000` 之前的命令视为测试条件；第一个 `0000` 之后到下一个 `0000` 之前视为事件体。
  - 若某事件直到 段落 结束都没有遇到 `0000`，则整段视为“事件体”。
- 子事件界面嵌套（启发式）：基于旧工具 `CczString.ini` 的先验知识：
  - `0001` = 子事件设定（子事件开始标志候选）
  - `0000` = 事件结束（用于关闭最近的子事件块；如果没有子事件在栈中，则作为事件体结束边界候选）
  - 目前的树展示为：`事件 -> 事件(体) -> 子事件 -> ... -> Command`
  - 注意：这仍是界面层启发式，参数长度与真实跳转语义未完全确认；遇到不成对标记时，界面 会尽量“就近闭合/终止”避免无限嵌套。

相关代码位置：

- 树构建：`CCZModStudio/MainForm.cs` -> `BuildScriptTree(...)`
- 段落/场景 文本归属与联动：`CCZModStudio/MainForm.cs` -> `BuildScriptSectionTextAssignments(...)`、`BuildScriptSceneFallbackTextAssignments(...)`、`GetScriptTextsForNodeContext(...)`
- 结构探针：`CCZModStudio/Formats/ScenarioStructureProbeReader.cs`
