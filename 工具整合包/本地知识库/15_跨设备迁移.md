# 跨设备迁移与路径解析

## 根因

- 旧逻辑把 `WorkspaceRoot`、游戏根目录和 `HexTable.xml` 绑定在同一个固定目录结构下。
- `HexTable.xml` 只查找：
  - `老版游戏制作工具\CczRSX 6.5\ConfigTable\HexTable.xml`
  - `CczRSX 6.5\ConfigTable\HexTable.xml`
- 跨设备复制、只复制游戏目录、从 `bin` 输出目录启动、或测试副本位于独立目录时，固定相对路径会失效，导致后续数据表、剧本和资源模块无法读取表结构。

## 当前规则

- 项目检测入口统一在 `CCZModStudio.Core.ProjectDetector`。
- 优先保留旧目录结构候选；失败后从以下起点向父级和有限深度子级搜索：
  - 用户选择的游戏目录
  - `Environment.CurrentDirectory`
  - `AppContext.BaseDirectory`
  - 已解析 `HexTable.xml` 所在目录
- `HexTable.xml` 可迁移到：
  - `<工作区>\老版游戏制作工具\CczRSX 6.5\ConfigTable\HexTable.xml`
  - `<工作区>\CczRSX 6.5\ConfigTable\HexTable.xml`
  - `<游戏目录或工作区>\ConfigTable\HexTable.xml`
  - `<游戏目录或工作区>\HexTable.xml`
- `CczString.ini`、素材库、B形象指定器、普罗搬运配置也应走 `ProjectDetector` 的 portable search API，不再各自硬编码两三个固定路径。
- `CczProject` 必须保存项目级已解析路径：`SceneDictionaryPath`、`SceneEditorDirectory`、`ImageAssignerDirectory`、`ImageAssignerSystemIniPath`、`MaterialLibraryRoot`、`PatchConfigRoot`。
- 主界面加载项目后应主动读取 `CczString.ini` 与素材库索引，而不是等到各模块首次使用时再各自猜测路径；手动按钮只作为刷新入口。
- 各模块读取旧工具资源时优先使用 `CczProject` 的已解析路径，并先校验 `File.Exists`/`Directory.Exists`；路径失效后再回退 portable search，避免旧设备绝对路径短路。
- 素材库索引、人物 R/S 形象预览、宝物大类边界、普罗补丁/搬运配置都应接收或引用 `CczProject`，不要只传 `WorkspaceRoot`。

## 受影响模块

- 数据表：`HexTableParser` 只解析传入 XML，不做路径猜测；路径统一由 `ProjectDetector` 解析。
- 剧本：`CczString.ini` 使用项目级搜索，支持旧剧本编辑器目录迁移。
- 人物 R/S 形象预览：新剧本编辑器配置和 B形象指定器配置使用项目级搜索。
- 宝物/物品大类：`System.ini` 从 B形象指定器目录迁移搜索。
- 素材库、普罗补丁/搬运配置：使用 portable directory search。
- 回归测试：`--migration-smoke` 应输出 `sceneEditor`、`imageAssigner`、`materialRoot`、`patchRoot` 与素材数量，并校验非空解析路径仍存在。

## 验证

- `dotnet run --project ".\工具整合包\CCZModStudio.SmokeTests\CCZModStudio.SmokeTests.csproj" -- --migration-smoke`
- `dotnet run --project ".\工具整合包\CCZModStudio.SmokeTests\CCZModStudio.SmokeTests.csproj" -- --rs-smoke`
- `dotnet run --project ".\工具整合包\CCZModStudio.SmokeTests\CCZModStudio.SmokeTests.csproj" -- --rs-write-smoke`
