## 详细记录

#### 概述

曹操传中的暴击（也称"必杀"）是一种在物理攻击时触发的高伤害特殊效果。当暴击触发时，攻击伤害会大幅提升。

暴击判定分为两层：
1. **基础暴击判定**：由 `0x426B33` 函数进行，基于攻击者状态和能力
2. **必定暴击特技**：当基础判定失败时，检查攻击者是否拥有"必定暴击"特技

#### 第一层：基础暴击判定函数 0x426B33

##### 函数反汇编

```asm
0x426B33: push ebp
0x426B34: mov ebp, esp
0x426B36: call 0x41B500           ; mov eax, [ecx+0x10] — 检查HP > 0
0x426B3B: test eax, eax
0x426B3D: jz 0x426B4E            ; HP=0则跳转, 返回al=0x02
0x426B3F: push 0x08               ; 参数 = 0x08（暴击标志位）
0x426B41: call 0x423FE8           ; 位匹配检查 unit[0x1E] & 0x08
0x426B46: test al, al
0x426B48: jnz 0x426B50           ; 匹配成功则跳转
0x426B4A: xor eax, eax            ; 匹配失败, eax=0
0x426B4C: jmp 0x426B50
0x426B4E: mov al, 0x02            ; HP=0, 返回0x02
0x426B50: pop ebp
0x426B51: ret
```

##### 子函数分析

**0x41B500** — HP读取函数（非特技检查）：
```asm
0x41B500: push ebp
0x41B501: mov ebp, esp
0x41B503: mov eax, [ecx+0x10]    ; 读取 unit[0x10] = 当前HP
0x41B506: pop ebp
0x41B507: ret
```

**0x423FE8** — 状态位匹配函数：
```asm
0x423FE8: push ebp
0x423FE9: mov ebp, esp
0x423FEB: push 0x00
0x423FED: movzx edx, byte [ecx+0x1E]  ; 读取 unit[0x1E]（状态标志）
0x423FF1: push edx
...
; 循环移位匹配：检查参数0x08是否在unit[0x1E]的位模式中
0x423FF5: cmp al, 0x04              ; 最多4次迭代
0x423FF9: shr [ebp+0x08], 1         ; 参数右移1位
0x423FFF: test al, 0x01             ; 检查最低位
0x424001: jz 0x42400D
0x424003: mov eax, [ebp-0x08]
0x424006: and eax, 0x03             ; 取状态的低2位
0x424009: test eax, eax
0x42400B: jnz 0x424018              ; 匹配成功
0x42400D: shr [ebp-0x08], 2         ; 状态右移2位
0x424011: inc [ebp-0x04]            ; 计数器++
0x424014: jmp 0x423FF2              ; 继续循环
0x424016: xor eax, eax              ; 匹配失败
```

##### 函数返回值

| 返回值 | 含义 |
|--------|------|
| `al = 0x00` | 暴击判定失败（HP正常但位不匹配） |
| `al = 0x02` | 暴击判定失败（HP=0，攻击者已阵亡） |
| `al = 非零` | 暴击判定成功（unit[0x1E]的位匹配0x08） |

#### 第二层：必定暴击特技 0x410F0A

当基础暴击判定（0x426B33）失败时，游戏会进行第二层判定——检查攻击者是否拥有"必定暴击"特技。

##### 在 0x410F1C 中的调用逻辑

```asm

0x410F23: call 0x4061F9            ; 获取攻击者指针
0x410F28: push ecx                 ; 保存
0x410F29: push 0x08                ; 参数=0x08
0x410F2B: call 0x423FE8            ; ★ 第一层: 基础暴击检查
0x410F30: test eax, eax
0x410F32: jnz 0x410F95             ; 基础判定成功, 返回
0x410F34: mov ecx, [ebp-0x08]
0x410F37: call 0x41932B            ; 防御/回避计算
...
0x410F50: mov ecx, [ebp+0x0C]     ; 防御者

0x410F58: test eax, eax
0x410F5A: jz 0x410F60             ; 失败则跳转到 ↓
0x410F5C: mov al, 0xFF             ; 成功, 返回0xFF
0x410F5E: jmp 0x410F97

; === 基础判定失败后的必定暴击特技检查 ===
0x410F60: mov ecx, [ebp-0x08]

0x410F68: test eax, eax

0x410F6C: mov ecx, [ebp-0x08]

0x410F82: test eax, eax
0x410F84: jz 0x410F97
0x410F86: push [ebp+0x0C]          ; 防御者

0x410F8C: call 0x4223A9            ; ★ 必定暴击执行
```

#### 暴击位的设置/清除机制

##### unit[0x1E] 是一个衍生字段

`unit[0x1E]` 位于**战术单位结构体**中（0x30字节结构，基址 0x4A7B20），而非角色单位结构体（0x61字节结构，基址 0x4A3E77）。该字段的值**不是直接写入**的，而是在战斗准备阶段从多个数据源聚合计算得出。

##### 两套单位结构体对比

| 属性 | 战术单位 (Tactical) | 角色单位 (Character) |
|------|-------------------|---------------------|
| 结构体大小 | `0x30` (48字节) | `0x61` (97字节) |
| 基址 | `0x4A7B20` | `0x4A3E77` |
| 指针公式 | `index × 0x30 + 0x4A7B20` | `id × 0x61 + 0x4A3E77` |
| 获取函数 | `0x4061F9` | `0x484002` |
| 内容 | 战场运行时数据 | 角色固有属性 |

##### 攻击者能力检查流程 0x43F8C0 详析

在物理攻击执行之前，0x43F8C0 会检查攻击者的各项状态：

```asm
; 0x43F8C0 — 攻击者能力与状态检查
0x43F8C4: call 0x40658F            ; 获取UI上下文
0x43F8D7: mov eax, 0x500E9A
0x43F8DC: cmp [eax], 0x02          ; 检查攻击类型=物理攻击
0x43F8DF: jnz 0x43F8F5
0x43F8E1: cmp [eax+0x01], 0x00     ; 检查 0x500E9B != 0
0x43F8E5: jz 0x43F8F5
0x43F8E7: mov cl, [eax+0x01]       ; 攻击子类型
0x43F8EA: call 0x41E918            ; 查表: cl*0x10 + 0x4A7510
0x43F8EF: mov al, [ecx+0x0C]       ; 读取攻击类型的标志字节
0x43F8F2: mov [ebp-0x08], al       ; 暂存到局部变量

; Mod引擎状态检查序列:
0x43F8FB: call 0x4D3242(0x05,0x00,0x32)  ; 检查状态A
         → 如果返回非全0 → [ebp-0x07] = al
0x43F90C: call 0x4D25B6(0x00, attacker)   ; 检查状态B  
         → 如果返回非0 → [ebp-0x06] = 0x00 (清除)
0x43F91E: call 0x4D25CB(0x00, attacker)   ; 检查状态C
         → 如果返回非0 → [ebp-0x05] = 0x00 (清除)
0x43F931: call 0x4E4000(al, attacker)     ; 遍历全单位检查
         → 修改 [ebp-0x08] (攻击目标)

; 最终检查: 攻击者是否被阻止攻击
0x43F939: push 0x04               ; ★ 检查 bit 0x04
0x43F93E: call 0x423FE8            ; unit[0x1E] 的 bit 0x04?
0x43F945: jz 0x43F94E             ; 如果未设置 → 攻击正常
0x43F947: or eax, -1              ; 如果已设置 → 攻击被阻止
0x43F94A: mov al, 0x0B            ; 返回 0x0B = 攻击不可执行
```

##### unit[0x1E] 已知位定义

| 位 | 参数 | 含义 | 检测位置 |
|----|------|------|----------|
| `0x01-0x03` | — | 状态类型/等级组合 | 0x423FE8 匹配 |
| `0x04` | `0x04` | **攻击阻止**（不可攻击） | 0x43F93E |
| `0x08` | `0x08` | **暴击状态**（可暴击） | 0x426B41, 0x410F2B |
| `0x10` | `0x10` | **中毒状态** | 0x423FA6 |

##### 状态位来源（推断）

基于代码架构分析，`unit[0x1E]` 的各位由以下来源聚合：

```
unit[0x1E] 状态字节的组装流程:

  ┌─ 装备/宝物特效 ─────┐
  │ (Mod引擎查表，        │
  │  如 0x35C0查表)      │──→ 0x4Dxxxx/0x4Exxxx
  ├─ 兵种固有特性 ──────┤    函数在攻击准备
  │ (角色数据查表         │    阶段计算并设置
  │  0x41B782:            │    各位标志
  │   id*0x23+0x4A2A27)  │
  ├─ 活跃Buff/Debuff ───┤
  │ (回合状态标志         │
  │  0x508B00, 0x501BF9) │
  └─ 攻击类型属性 ──────┘
  (0x41E918 查表:
   0x500E9B*0x10+0x4A7510)
```

##### 暴击位 (0x08) 的设置来源（推断）

| 来源 | 触发条件 | 清除条件 |
|------|----------|----------|
| 兵种特性 | 特定兵种天然暴击 | 无（永久） |
| 武器/宝物 | 装备暴击加成物品 | 卸下装备 |
| 攻击类型 | 特定攻击指令自带暴击 | 攻击结束 |
| Buff/法术 | "奋起"/"会心"类法术 | 回合结束/被解除 |

##### 静态分析瓶颈

当前纯静态分析无法确定 `unit[0x1E]` 的写入代码，原因：
1. Mod引擎函数（0x4Dxxxx/0x4Exxxx）深度封装，调用链复杂
2. 状态位可能由 0x4E4000 内部的 0x4EE0C8 → 0x4EE09C/0x4EE060 子函数写入
3. 也可能在战斗初始化阶段（0x44AF8F 回合重置循环附近）预设

**建议**：在 x64dbg 中对战术单位的 `[unit_ptr + 0x1E]` 内存地址设置**硬件写入断点**，然后执行一次物理攻击，即可直接捕获写入点。

#### 与连击机制的关系

暴击和连击是独立的判定：
- 基础暴击：通过 unit[0x1E] 位匹配 + HP检查
- 必定暴击：通过特技桩→调度→核心引擎
- 连击：通过"爆发力"比对（见 double-attack.md）
- 两种效果可以同时触发，独立计算

#### 相关函数索引

| 地址 | 函数 | 功能 |
|------|------|------|
| 0x426B33 | 基础暴击判定 | HP检查 + 状态位匹配 |
| 0x41B500 | HP读取 | 读取 unit[0x10]（当前HP） |
| 0x423FE8 | 状态位匹配 | 检查 unit[0x1E] 的特定位模式 |
| 0x410F0A | **必定暴击桩函数** | 特技桩（内联在战斗代码中） |
| 0x60031A | 必定暴击调度函数 | EDX=0x35C0, 调用核心引擎 |
| 0x4101D9 | 核心引擎 | 执行特技效果 |
| 0x4223A9 | 必定暴击执行 | 设置0x505EFC, 播放动画 |
| 0x410F1C | 伤害计算主函数 | 包含完整暴击判定流程 |
| 0x41932B | 防御计算 | 计算回避/防御效果 |
| 0x41CECC | 暴击伤害计算 | 计算含暴击的伤害值 |
| 0x412B30 | 防御修正 | 必定暴击时的额外防御计算 |
| 0x4363DC | 伤害应用 | 应用暴击伤害 |

#### 设计注意事项

如果要实现自定义暴击系统：

1. **基础暴击修改**：修改 `0x426B33` 中的判定逻辑
2. **必定暴击扩展**：在 `0x410F0A` 的桩函数模式中添加新的特效号
3. **必定暴击效果**：修改 `0x4223A9` 中的特效播放
4. **状态位**：`unit[0x1E]` 是暴击判定的核心状态字段
5. **调度表**：`0x35C0` 是必定暴击在调度表中的偏移
