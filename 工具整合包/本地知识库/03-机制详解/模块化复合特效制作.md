# 模块化复合特效制作

## 设计原则

模块不是任意文字标签，而是带输入、输出、值域、Hook 契约、冲突组和证据 SHA 的类型化能力。扫描归纳和新增复合特效使用同一模块目录。

模块类别包括：触发时机、判定对象、作用目标、生效条件、实际效果、数值方式、配置来源和安全契约。

“使用角色”被拆成两层：

- 判定对象和作用对象：运行时 Hook 中的单位指针。
- 配置来源：绑定到武将、兵种、人物专属、套装或物品的数据表记录。

二者不能互相替代。

## 首版开放边界

- “复合现有特效”配方可以生成注入包，至少选择两个兼容成员。
- 兼容成员必须属于直接核心调用、已验证包装链或已验证复杂家族。
- 策略伤害调整已经具有受约束业务编译器，能够生成固定值或当前伤害百分比的增减伤代码；但当前 SHA 的 `0043C2B0-0043C2C8` 是连续填充区，尚未证明所有策略伤害路径都会自然执行，因此执行契约仍处于“静态候选”，只能生成诊断，不能写入。
- 物理伤害后恢复、异常状态、行动结束和回合触发仍缺当前 SHA 的完整动态契约，只用于扫描标签和待验证队列。

## 蓝图和注入

`ModularCompositeEffectBlueprint` 保存配方、渠道、新编号、中文名称、模块稳定编号、成员参数和绑定。“复合现有特效”转换为 `CompositeEffectDraft`；受约束动作转换为 `SemanticEffectProgram`，再由语义编译器生成业务代码。两条路径都必须经过代码洞、旧字节锁、事务备份和 manifest。

注入后的 manifest 保存蓝图 JSON。重新扫描时必须能恢复复合特效成员、参数块和模块标签；不能往返识别时显示安装不完整。

## MCP 顺序

1. `list_effect_modules`
2. `read_effect_module`
3. `validate_effect_blueprint`
4. `preview_modular_effect`
5. 用户复核后显式调用 `apply_modular_effect`

任何模块不可用、值域不符、上下文缺失或冲突组不一致都会返回中文阻断原因，不能强制绕过。

## 执行契约与共享调度

Hook 地址本身不代表可以注入。`HookExecutionContract` 还必须证明判定对象、作用目标、当前伤害或恢复字段、原逻辑顺序、上下限顺序、寄存器、标志位和栈平衡。

动态验证流程：

1. `read_hook_execution_contract` 读取当前 SHA 的槽位和缺失证据。
2. `create_effect_probe_session` 由 GameDebug 为临时副本创建会话，`run_effect_probe_batch` 按批布置断点，用户完成指定战斗动作后调用 `capture_effect_probe_scenario`。
3. 四类场景完成后由 `finalize_effect_evidence_bundle` 签发 `EffectEvidenceBundleV1`；证据绑定进程、加载模块路径/大小/SHA、原始文件摘要、特效能力 schema、构建通道和当前 Windows 用户 HMAC。
4. Core 只通过 `import_effect_evidence_bundle` 接受可从原始 JSON 复算的观测。旧 `import_effect_contract_evidence` 只保存诊断材料，不能提升写入能力。
5. 普通样本的观测最小值/最大值不等于引擎合法边界。可写槽位必须从边界场景单独证明合法上下限，才能开放算术调度器。

同一触发契约最终使用一个 `CCZD v2` 共享调度器。调度器已实现真实运行时循环，调度注册表容量按 `16 -> 32 -> 64` 扩展，每条记录保存启用状态、双渠道编号、判定参数、动作、数值来源和执行顺序。同一触发点第二个特效只更新注册表，不重复安装入口 Hook。

调度器 manifest 只拥有入口、代码体和 `CCZD` 注册表，不拥有首个特效的名称、说明或绑定。已有调度器追加条目时，Core 必须同时复读入口、代码体、注册表、安装时执行契约快照和原始动态证据；旧清单没有契约快照时保持只读。

当前策略伤害调度器已修正固定值/百分比增减伤和契约上下限钳制，但当前 SHA 尚无合法的四场景可信证据包，所以普通界面仍显示“研究中”。恢复动作还缺当前值/最大值成对槽位及界面刷新顺序，编译器和调度器均显式阻断。

## 写入凭据与发布一致性

- 原生配置、参数适配器、汇编补丁、复合特效、模块化特效及维护操作统一使用当前进程签发的短期一次性凭据。包元数据、绑定、写入段、旧字节、项目身份或 SHA 任一变化都会失效。
- 正式发布使用 `effect-release-manifest.json` 锁定桌面/Core、Runtime、MCP 和 GameDebug 的文件版本、构建标识与 SHA。清单存在但组件不一致时，桌面和 MCP 都禁止写入，预览仍可查看。
- `MCP配置/publish-effect-release.ps1` 只向独立暂存目录构建并生成待切换发布；`activate-pending-effect-release.ps1` 只在桌面、MCP 和 GameDebug 全部退出后执行目录级切换。

## 参数宽度

`push imm8` 只能安全表达 `00-7F`。修改为 `80-FF` 时，`preview_effect_id_update` 会尝试生成宽参数适配器：选择完整指令窗口、检查中间入口、用 Iced 搬迁其余指令，并以 `push imm32` 替换原参数。若存在共享入口、分支或不可重定位指令，保持只读。

## 2026-07-16 物理攻击后恢复 MP 试点

- 契约 ID 保持 `physical-after-damage-recovery-v2`，版本升为 3，旧契约哈希和旧证据自动失效。
- Hook 锁定 `00418335` 原有 rel32；新调度器先执行，保存/恢复 EFLAGS、通用寄存器、EBP/ESP，尾跳 `004528FC` 继续遗留 `C0/00` 回 MP 链。
- 只开放 `RestoreMpFixed=5`、人物/兵种渠道、宝物号 `00`、叠加方式 `1`；上限来自 `physicalContext+08 -> RuntimeCharacter+1F WORD`，当前 MP 来自 `physicalContext+0C -> TacticalUnit+14 DWORD`。
- V3 配方使用 `00418335/004528FC/0045290A/00452975/00405AD5/0043F70C` 六个独立提取器，覆盖普通、暴击、反击、连击、MP=0、接近最大值和负向七场景；连击各关键点要求两次命中。
- 物理契约的 `EvidenceDisposition=ValidationOnly`。`validate_effect_evidence_bundle_v3` 只读验证签名、原始文件、基础/沙箱 SHA、补丁包、探针计划、关系、边界和 continuation；`VerifyAndStoreV3` 明确拒绝入库。
- 安装只允许自动签名副本的 `SandboxValidation`。正式项目仍无行为逻辑 Apply 权限。
- 当前规范库存 `01-FE` 没有完全空白人物/兵种号，只有两个“仅名称可回收”号；试点不借用它们，因此实际动态七场景仍等待释放专用测试号。
