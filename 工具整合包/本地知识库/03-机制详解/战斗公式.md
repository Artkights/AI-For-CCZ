#### 数据结构基础

##### 关键单位属性偏移

| 偏移 | 类型 | 属性 | 读取函数 |
|------|------|------|----------|
| `unit+00` | 字 | Data 号 | 40658F/4061E4 |
| `unit+04` | 双字载体 | 出场/显示编号 | UI 更新调用点；不是 Data 号 |
| `unit+10` | 双字 | 当前 HP | 41B500/43F70C |
| `unit+14` | 双字 | 当前 MP | setter 待动态闭环 |
| `unit+1E` | 字节 | 异常状态组合 | 423FE8 等 |
| `character+11/+13` | 各字 | 基础攻击/防御 | 人物运行时记录，非战术单位 |

##### 指针计算公式

```
战术单位指针 = 4A7B20 + 战场编号 × 30H
人物运行时指针 = [4CEA00] + Data号 × 48H
策略记录指针 = 4A3E77 + 策略号 × 61H
```
其中人物记录由 `4061E4` 取得；`484002` 只用于策略记录：
```asm
484005: imul ecx, ecx, 61       ; 61H，即 97 字节
484008: add ecx, 4A3E77
```

#### 二、伤害计算公式

##### 基础判定：攻防比对

**核心逻辑** 位于 `417B27` → `41CECC` 调用链：

```
伤害有效条件: 攻击力 ≥ 有效防御力
```

具体实现：

```
Step 1: 读取攻击力
   attack = [attacker_unit + 14]  (word, via 472840)

Step 2: 读取防御力
   defense = [defender_unit + 0E]  (byte, via 425AD0)

Step 3: 穿透修正 (如果攻击者有破防能力, via 44DF49)
   if has_penetration:
       ratio = 100 - penetration_value
       defense = defense × ratio / 100  (via 41EDC1)

Step 4: 最终判定
   if attack >= adjusted_defense:
       攻击命中，产生伤害
   else:
       攻击无效
```

##### 关键计算公式

**百分比缩放函数** [41EDC1](../03-机制详解/战斗公式.md)：

```asm
41EDC4: movzx ecx, cl      ; cl = 百分比(0-100)
41EDC7: imul eax, ecx      ; eax = value × ratio
41EDCC: mov cl, 64       ; cl = 100
41EDCE: div ecx            ; eax = (value × ratio) / 100
```

即：`result = value × ratio / 100`

##### 伤害公式（简化）

```
基础伤害 = max(攻击力 - 有效防御力, 0)

其中:
  有效防御力 = 防御力 × (100 - 破防值) / 100
              (无破防能力时破防值=0，即有效防御力=防御力)
```

##### 暴击伤害加成

当暴击判定成功时（详见第三节），伤害增加：
- 在 `4179B2` → `425627` 中处理暴击伤害倍率
- `00497AF8` 是策略上下文基址；本条调用链若借用其字段参与倍率，必须按具体偏移解释，不能把整块结构命名为暴击加成系数

#### 三、暴击率计算公式

##### 暴击判定机制

暴击不是概率判定，而是**状态标志位判定**。核心函数是 `423FE8`。

```
单位状态字段 unit[1E] 是一个 8-bit 打包标志位:
  
  bit pattern: [??][??][??][??][??][??][01][00]
                                  ↑
                             暴击标志 = 08 (bit 3)

当 unit[1E] 的位匹配参数 08 时，单位处于"可暴击"状态。
```

##### 判定函数 [423FE8](../03-机制详解/战斗公式.md)

```
参数: 状态位掩码 (如 08 = 暴击)

流程:
1. 读取 [unit+1E] → 获取单位状态字节
2. 循环匹配 (最多4次迭代):
   a) 参数右移1位，检查最低位
   b) 如果最低位为1，取状态的低2位
   c) 如果状态低2位匹配，返回成功
   d) 否则状态右移2位，继续循环
3. 4次迭代后仍未匹配 → 返回失败
```

##### 完整暴击判定流程

```
┌─────────────────────────────────────┐
│ 1. 攻击者HP > 0?  (41B500)       │
│    ↓ 是                            │
│ 2. unit[1E] 位匹配08?          │
│    (423FE8)                       │
│    ↓ 是                            │
│ ★ 暴击成功 → al=FF               │
└─────────────────────────────────────┘
```

##### 暴击触发条件

| 条件 | 检查方式 | 说明 |
|------|----------|------|
| 攻击者存活 | `[unit+10] != 0` | HP必须>0 |
| 暴击状态 | `unit[1E] 含 08 位` | 单位必须具有暴击状态 |

> **注意**：当 unit[1E] 的暴击位被设置时，该单位的每次攻击都必定暴击。
> 在常规流程中，暴击位的设置/清除受装备、状态、兵种特性影响。

##### 当前分析状态

命中/回避判定尚未在当前分析的代码路径中直接定位。根据游戏机制推理：

```
待证假设公式:
  命中率 = 攻击方命中值 - 防御方回避值 + 基准值
  
  其中:
    - 命中值/回避值 = 敏捷属性 × 兵种系数 + 装备加成
    - 基准值 = 100 (or other constant)
    - 结果 clamp 到 [0, 100] 范围
```

#### 五、连击率计算（待深入验证）

##### 当前分析状态

连击（double attack）基于**爆发力比对**的机制尚未在当前分析的代码路径中定位。

根据用户描述和曹操传经典机制：
```
连击率 = (攻击方爆发力 - 防御方爆发力) × 系数

其中:
  爆发力 ≈ 敏捷属性 + 装备/兵种加成
  系数 = 触发连击所需的差值比例
```

##### 已定位的相关代码

在 `43DADA` 攻击流程末尾，`43DB80` 调用 `4212F1`：

```
43DB80: call 4212F1    ; 追击/连击检查
```

但 `4212F1` 内部首先调用的是特技桩函数 `4210D0`（装备号=36，个人号=18 → 调度函数6002C6），这是**追击特技**，而非基础连击机制。

基础连击（爆发力比对）的计算位置还需进一步在以下区域搜索：
- `43F8C0` 攻击者能力检查（可能在内部进行爆发力比对）
- `4D3000` 开始的 模组引擎函数（可能在内部计算连击率）
- 或战斗准备阶段（非攻击执行阶段）的预计算

#### 2026-06-11 测验1动态确认：最终扣血落点与公式边界

证据等级：x32dbg live 命中 + HP 写断点闭环。场景为曹操攻击黄巾兵，用户确认两次 HP 写入来自连击。

本轮只确认最终扣血落点：

```text
00405966  攻击结果/伤害结果循环
    -> 00405ADC
        -> 0043F6DB  HP setter
            -> 0043F70C  mov [ecx+10], edx
```

本轮实际扣血：

| 段数 | 旧 HP | 新 HP | 推得伤害 |
|------|-------|-------|----------|
| 第 1 段 | `118` | `59` | `59` |
| 第 2 段 | `59` | `15` | `44` |

边界：

- `0043F70C` 只是最终 HP 写入，不是伤害公式。
- `00405966` 使用已经生成的攻击结果/伤害记录并循环应用，不应直接标记为“伤害计算函数”。
- 本次没有动态闭环 `59` 和 `44` 这两个数值的上游来源；攻防、地形、兵种相克、随机浮动、连击第二段修正等因素仍需反向追踪。

推荐下一步：

- 在 `00405966` 入口采集结果缓冲区和栈变量，追踪 `[ebp-14]`、`[ebp-18]` 等伤害相关临时值来源。
- 对疑似结果缓冲区（例如 `004927F0` 附近及其动态引用）设置写断点，找到 `59`、`44` 被写入的位置。
- 在 `0043DADA` 内部对伤害记录生成分支做单步/条件断点，区分普通一击、连击、反击、暴击的结果记录生成逻辑。
