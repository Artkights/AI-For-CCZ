# 通用曹操传 MOD 剧本提示词模板

本模板用于让 AI 为曹操传系 MOD 生成高质量剧本文案和战场设计。它不是某一条固定主线的专用模板，不默认刘备军、曹操军、姜维、蜀汉传或任何固定主角。

主线由输入决定：`MOD名称`、`主线阵营`、`主角/视角人物`、`时代节点`、`世界观`、`史实/架空程度`、`当前战役视角`。如果输入不足，AI 必须先从剧情描述推断；仍无法判断时输出“缺少主线设定”，不得擅自套用魏、蜀、吴或姜维传视角。

## 样本经验：6.5 神话三国志前三关 R/S

已参考的样本为 6.5 未加密基底中 `RS/R_00.eex`、`RS/S_00.eex`、`RS/R_01.eex`、`RS/S_01.eex`、`RS/R_02.eex`、`RS/S_02.eex`。这些不是原版《曹操传》的传统前三关，而是 6.5 基底《神话三国志》的开局样本；后续生成只学习结构、节奏和实现方法，不复刻原台词，不默认原版魏线或任何固定主角。

### 0. 首关 R_00 与独立基底规则

- 6.5 原版 `RS/R_00.eex` 是开局启动容器，结构复读为 `5 Scene / 7 Section`，承担初始化、历史/指南、模式选择和正式剧情入口；它是样本事实，不是独立精简基底必须保留的实现形态。
- 独立基底首关如果只服务单关 / 新主线，应采用 active `R_00.eex / S_00.eex` 的纯净流程：`R_00` 不保留原版启动、菜单、随机、模式、配置分支；剧情正文只保留剧情、人物出场、停顿、入队、装备、物品获得和出战设置。文学分幕只能写成正文内的 `0x16` 标题卡、`0x17` 场所名、`0x18` 事件名和 `0x09` 停顿，不要创建多个真实剧情 Section。
- `0x11 剧本跳转` 在本地 6.5 复读中的目标映射为：`0 -> S_00.eex`、`1 -> R_01.eex`、`2 -> S_01.eex`、`3 -> R_02.eex`。这只适用于明确采用硬跳转路线的脚本；当前已实机验证的首关战前准备路线不在 R 侧写 `0x11`，而用双 Scene 的 `0x07/0x08/0x06/0x1D/0x0D` 收束。
- 需要战前出战入口时，优先采用本地样本式双 Scene 流程，而不是 `07 + 11` 硬跳：剧情 Scene 末尾写 `0x4B` 我军候选出战位、`0x08 true`、`0x06 出场限制`；下面另起玩家切换 Scene，写 `0x07 出战测试 -> 0x00 -> 0x08 false -> 出发对白 -> 0x06 默认/清理 -> 0x1D -> 0x24 255 -> 0x0D -> 0x00`，由引擎进入对应 S。`0x72 信息传送` / `4051` 缺少本关可复用首关样本，且《三英战龙帝》实机反馈曾出现剧情跳跃和进不了 S 的问题；自动化生成不得把 `0x72` 作为默认首关战前准备模板。
- 独立首关 `S_00` 默认优先复读并使用 `M000` 地图槽；当前 6.5 布局中 `0x44 战斗初始化` 为零参数，不要在 `0x44` 里填写地图号。若设计需要 `M056` 等大地图，必须先验证脚本编号到地图槽的绑定，不能只改坐标。
- `0x4A` 是强制出战 / 禁用列表，首值为强制出战人数，后续为人物 Data ID；`0x4B` 是我军候选出战位，参数为 `出战顺序 / X / Y / 方向 / 标志`，第一个参数不是人物号。强制人物按 `4A` 顺序绑定到 `4B` 候选位顺序。
- 生成 S 事件时，`0x05 变量测试` 必须按旧版完整树的 VariableArray 参数写入，不能当普通整数参数拼接。地图坐标必须来自已复读的 `Map/Mxxx.JPG` 尺寸与 Hexzmap 块，例如 `M000=960x960=20x20`。
- 写入或生成后必须复读：独立首关 `R_00` 不含 `0x12/0x13/0x75/0x76/0x77` 等基底启动命令，不再出现“历史情报/游戏指南/开局设定”文本；若采用当前推荐战前入口，结构应为剧情/出战设置 Scene + 玩家切换 Scene，且不含 `0x11` 硬跳。`R_00` 内禁止 `44/45/46/47/48/4A/4C/4D/4E/4F/50/53/55/58/5A/5D` 等 S-only 命令，`4B` 只允许作为 R 侧候选出战位。
- `S_00` 进入后闪退时不要先回滚 R。先按 S 样本链检查：`44 战斗初始化 -> 45 战场全局变量 -> 46/47 部署 -> 4B 我军位 -> 48 装备 -> 27 背景显示 -> 1C 绘图 -> 24 音乐 -> 19/1A 胜败条件 -> 5A 战场操作开始 -> 0B 全局进度 -> 08 菜单处理`。缺 `45/27/1C/08` 或部署、装备数组不完整，优先判为 S 初始化问题。
- R 场景坐标不能复用 S 战场坐标；`0x30/0x32` 必须按 `pixelX = 8 * (x - y + 42)`、`pixelY = 4 * (x + y - 50)` 校验可见。对白之间用 `0x09 延时` 分节奏，避免连续平铺。

### 1. R 剧情的有效结构

- R 剧情不只写对白，必须承担“战后复盘、因果变化、人物关系推进、系统说明、跳转 S”的桥接职责。`R_01` 的战后复盘先解释上一战结果，再用“主角介入导致历史偏移”承接世界观，随后用人物互动引出召唤系统和下一战。
- 系统说明不能脱离剧情硬讲。样本会把召唤、天赋、历史平衡、角色加入等规则放进人物对话，但关键数值、限制、奖励和触发点仍要在战前情报中明确列出。
- 每个进入 S 的关卡都应给玩家一张“战前情报卡”，至少列出单挑、关键对话、物品坐标、特殊事件、胜败风险。情报卡可放在 R 结尾，也可因文本框节奏放到 S 开场；放到 S 时，R 结尾只保留入阵和跳转。

### 2. S 战场的有效结构

- S 开局先做地图、全局变量、出场、装备、能力和特技等初始化，再显示胜败条件与玩家可见目标。生成时必须区分“显示给玩家看的目标文本”和“实际判定事件”。
- 目标可以在战中变更。`S_00` 先要求救出关键人物，触发对话后改为全灭敌军，并追加回合上限；模板必须要求写清楚“初始条件、变更触发、变更后条件、失败条件变化”。
- 单挑、对话、物品、阶段推进都要落成可执行事件，而不是只写在剧情里。生成前必须查 `08-剧本与战场/剧本设计语义命令映射表.md`，把设计语义写成真实命令映射；例如单挑触发优先落到 `0x2E 武将相邻测试`，奖励优先落到 `0x3D 获得物品`。具体 S 关卡中的关键对话有触发顺序要求，特定敌人只能按指定方式击败；这类限制必须进入事件图。
- 多阶段地图要明确阶段门。`S_02` 同时包含逃脱目标、王越相关事件和后续多波区域推进，说明 R 的战前情报、S 的胜败条件、战中事件可能分散在不同段落；生成和修复时必须通读命令树核对，而不能只看标题或单条文本。
- 隐藏奖励必须写成坐标、访问者、触发次数、奖励内容和提示文本；村庄、兵营、宝物库、坐标点、单挑奖励都要进入同一份奖励清单。

### 3. 可复用制作原则

- 先定结构，后写台词：每关先输出关卡目标、阶段、事件图和奖励表，再生成 R/S 文案。
- R 负责“为什么打、打前知道什么、打完世界如何变化”；S 负责“怎么打、何时变更、如何判胜败、如何发奖励”。
- 自动化生成必须产出机器可读的 `ScenarioDesign` / `ModPackage`，不能只交自然语言剧本。
- 任何 R/S 指向、标题、胜败条件、坐标、人物 ID、变量、跳转编号存在不一致时，先列为“待核验项”，不得为了顺滑剧情自行编造。

## 一、通用场景过场剧本提示词

适用于 R 场景、战前/战后过场、章节剧情、人物关系推进。把下面整段复制给 AI，并替换尖括号内容。

```text
你是一位精通《三国志曹操传》与曹操传系 MOD 剧本写作的资深文案，并能达到《姜维传》的人物刻画水准。请严格按以下要求，将给定剧情改写为曹操传风格的过场剧本。

【主线设定】
- MOD名称：<MOD名称>
- 主线阵营：<魏/蜀/吴/晋/群雄/原创势力/多主角群像/其他>
- 主角/视角人物：<本段剧情的主视角人物；群像时写本幕核心视角>
- 时代节点：<具体年份、战役阶段或章节位置>
- 世界观：<史实向/演义向/架空改写/幻想扩展/原创外传>
- 史实约束：<必须遵守的历史事件、人物关系、地理边界>
- 人物存亡约束：<当前时间点已死亡、不可登场、可回忆提及的人物>
- 剧情梗概：<需要改写的剧情内容>

【主线判定规则】
1. 用户明确给出主线阵营、主角或当前视角时，严格服从。
2. 用户只给剧情时，以剧情中当前可控方、叙事焦点或情感焦点作为本段主线。
3. 多主角或群像 MOD 中，只确定“本段/本关视角人物”，不得改写全局主线。
4. 信息不足且无法可靠推断时，最终只输出：缺少主线设定：请补充主线阵营、主角/视角人物、时代节点。
5. 不得自动假定刘备军、曹操军、姜维、蜀汉传或任何固定势力为默认主线。

【结构要求】
- 将剧情拆分为若干幕。每一幕必须有标题，格式为“第X幕 地点·场景”，如“第一幕 汉中·军营”。
- 各幕之间空一行。
- 第一行必须直接是场景标题，不输出说明、分析、清单或 Markdown 代码围栏。
- 如本 R 承接上一战，必须按“战后复盘 -> 因果变化 -> 人物关系推进 -> 系统/规则说明 -> 下一战战前情报 -> 跳转 S”的顺序组织；不需要的环节可以省略，但不得颠倒因果。
- 如本 R 引出下一场 S 战，结尾必须包含一段玩家可见的“战前情报卡”，用剧内信息或系统提示列清单挑、关键对话、物品、特殊事件、胜败风险。

【表现形式】
- 纯对话形式，禁止叙述性描写。
- 对话格式固定为：`【角色名】对话内容`。
- 动作、神态、镜头转换使用半角括号，可插入角色名与台词之间，也可单独成行。
- 允许必要旁白交代时空背景，格式为 `【旁白】内容`，但必须克制，不得代替人物对话推进剧情。

【人物塑造】
- 以“姜维传水准”作为人物刻画质量标准，而不是固定采用姜维或蜀汉视角。
- 每个人物必须有清晰性格底色和情感逻辑。
- 通过迟疑、停顿、回避目光、握紧兵器、压低声音等细节体现内心，不写空泛标签。
- 正邪立场之外，要体现恐惧、愧疚、温情、偏执、疲惫、不忍等人之常情。
- 关系变化要有层次：信任到猜忌、敌对到惺惺相惜、忠诚到动摇，都要体现在台词互动中。

【语言风格】
- 对白古朴简练，符合汉末三国语境。
- 情绪高点可以爆发，但避免现代口语、网络语和过度华丽的空话。
- 台词服务人物与冲突，不用旁白替角色解释内心。

【逻辑与时间线】
- 动笔前自动梳理时间节点、人物存亡、阵营关系和地点。
- 已死亡角色不可真实登场；如需出现，只能作为回忆、梦境、遗书、传闻，并必须在台词中明确性质。
- 不得让事件顺序、地理移动、人物关系与给定设定冲突。
- 模糊处选择最合理时间点，并用克制旁白明确。

【R/S 衔接要求】
- R 中出现的下一战标题、编号、参战人物、单挑、关键对话、奖励坐标、特殊机制，必须能在后续 S 设计中逐条落实。
- 系统教程、召唤、天赋、阵营变化、人物加入、装备奖励等内容必须同时写清“剧情解释”和“玩法效果”。
- 不复制样本台词，只学习样本结构。不得默认本关属于原版曹操传前三关、魏线、蜀线或任何固定势力。

【最终输出】
- 只输出剧本正文。
- 第一行直接以“第一幕 地点·场景”开始。
- 不附加解释说明。
```

## 二、通用战场剧本设计提示词

适用于 S 战场设计、单关设计、战斗事件规划。`我军` 表示本关玩家可控或剧情主视角阵营，不固定为刘备军。

```text
你是一位精通《三国志曹操传》与曹操传系 MOD 制作的资深战场剧本设计师。请严格按以下格式，将给定战役描述转化为完整战场剧本。

【主线设定】
- MOD名称：<MOD名称>
- 本关标题：<战役名>
- 主线阵营：<魏/蜀/吴/晋/群雄/原创势力/多主角群像/其他>
- 本关我军定义：<本关玩家可控方或剧情主视角阵营>
- 主角/视角人物：<本关主视角人物>
- 时代节点：<具体年份、战役阶段或章节位置>
- 世界观：<史实向/演义向/架空改写/幻想扩展/原创外传>
- 史实约束：<必须遵守的历史事件、人物关系、地理边界>
- 人物存亡约束：<当前时间点已死亡、不可登场、可回忆提及的人物>
- 战役剧情描述：<需要改编的战役剧情>

【主线判定规则】
1. `我军` 是本关玩家可控或剧情主视角阵营，不默认等于刘备军、曹操军或任何固定势力。
2. `友军` 是与本关我军同目标但不可控或临时协同行动的部队。
3. `敌军` 是本关胜利条件或剧情冲突中的对立阵营。
4. 用户明确给出主线时严格服从；只给战役描述时，以当前可控方/叙事焦点作为本关主线。
5. 多主角或群像 MOD 中，只确定本关视角，不改全局主线。
6. 信息不足且无法可靠推断时，最终只输出：缺少主线设定：请补充本关我军定义、主角/视角人物、时代节点。

### 二、战前情报卡与实现摘要

先输出本关玩家可见情报，再输出实现摘要。战前情报卡必须明确放置位置：R 剧本结尾或 S 开场战前说明；若 R 文本框节奏已经很密，优先放到 S 开场。

| 项目 | 内容 |
| --- | --- |
| 本关标题 | <S_编号 + 战役名> |
| 初始胜利条件 | <玩家进入战场后第一眼看到的胜利条件> |
| 初始失败条件 | <主角死亡、关键人物死亡、回合上限等> |
| 单挑 | <角色A VS 角色B，奖励或撤退效果> |
| 关键对话 | <角色A 与 角色B，对话效果、是否有顺序要求> |
| 物品与坐标 | <坐标、访问者限制、奖励内容> |
| 特殊事件 | <伏兵、状态变化、目标变更、阶段推进、无限复活等> |
| 待核验项 | <缺少 ID、坐标、地图、变量、跳转编号时列出，不得编造> |

实现摘要必须列出：
- 显示给玩家的目标文本。
- 实际判定胜利/失败的事件。
- 胜败条件是否会变更，变更发生在哪个事件后。
- R 剧本中承诺的单挑、对话、奖励、特殊机制分别由哪个 S 事件实现。

### 四、剧情节点设计

#### 1. 战前剧情（S剧本对话）
```text
回合开始前的各方势力对话剧情，纯对话模式。
```

#### 2. 开场白（战场加载后第一段对话）
- **触发条件**：战斗开始，第一回合开始前
- 格式同上，通常为简短战场态势对话或主将临阵激励。

#### 3. 战斗中触发事件

按以下格式逐条列出，每条包含触发条件与对话：

- **事件1：[事件名称]**
  - **触发条件**：[如：第3回合开始/敌方主将HP低于50/我方部队进入坐标(12,15)区域]
  - **前置条件**：[如：某变量未开启/某角色存活/某对话尚未触发]
  - **对话**：
    ```text
    【角色名】对话内容
    【角色名】(动作) 对话内容
    ```
  - **执行效果**：[出场、撤退、移动、变更阵营、改变 AI、加状态、加能力、给物品、设置变量等]
  - **验收点**：[玩家能看到什么变化，如何确认事件已触发]

必须根据战役剧情合理包含以下类型事件：
- 单挑事件
- 伏兵出现
- 倒戈/叛变
- 计策发动（火计/水计/落石等）
- 关键人物阵前对话

#### 4. 击败特定敌将的台词

逐条列出：
- **击败[敌将名]时**：
  ```text
  【敌将名】遗言……
  ```
  或
  ```text
  【我方角色】(收枪) 台词……
  ```

#### 5. 增援/撤退事件

- **事件1：[名称]**
  - **触发条件**：[如：第5回合/敌方兵力低于一定数量]

## 三、文案到可导入文本的三段式流程

1. 第一段：生成可读剧本文案。R 写过场、关系、系统说明和入场理由；S 写战前情报、开场、战中、战后、失败分支。
2. 第二段：生成结构化实现稿。必须输出 `ScenarioDesign`：R/S 编号、地图、出场、阶段、胜败条件、事件图、奖励清单、变量清单、待核验项。
3. 第三段：使用 `08-剧本与战场/剧本文本导入AI说明模板.md`，把已定稿文案和结构化实现稿转换为 CCZModStudio 文本导入格式。
4. 转换阶段不得继续扩写剧情；只能把已有台词、旁白、出场、移动、转向和动作转换为 `@dialog`、`@narration`、`@appear`、`@move`、`@turn`、`@action`、`@battle-turn`、`@battle-action`。
5. 缺少人物 ID、坐标、方向、动作编号、变量编号、S 跳转编号或战场触发条件时，转换阶段必须输出“缺少信息”，不得编造。
6. 自动化写包前必须先过一遍 R/S 对照表：玩家可见情报卡中的单挑、关键对话、物品、事件，在 S 事件图中逐条存在。

```text

- 主角/视角人物：<本关主视角人物>

## 五、后续单关 R/S 与全自动 MOD 的实行方案

0. 查自动化制作规则：写包前必须先读 `06-项目与工具链/自动化MOD制作.md`，确定是原基底追加、独立单关，还是独立精简基底；独立精简基底才允许清旧表信息和低位重占用。
1. 读样本：每次生成前先读取同类型样本 R/S，至少抽取“R 情报卡、S 胜败条件、事件图、奖励坐标、阶段推进”五类信息。
2. 定设计：先生成 `ScenarioDesign`，确认主线阵营、本关我军、角色、地图、初始目标、变更目标、阶段门、奖励表和待核验项。
3. 锁槽位：在设计稿中锁定 R/S、人物 Data、职业、武器、防具、辅助、奖励道具、变量、地图和资源门控；槽位冲突只能回改设计稿，不能写包时临场换号。
4. 写 R：按“背景/复盘 -> 冲突 -> 人物关系 -> 系统说明 -> 入阵收束 -> 跳转 S”写，不把机制藏在台词里；长战前情报可移到 S 开场。
5. 写 S：按“初始化 -> 部署 -> 显示目标 -> 事件图 -> 胜败判定 -> 阶段推进 -> 奖励 -> 胜败分支”写，所有机制都要有触发和防重复。
6. 命令映射：每个 S 事件都要查 `08-剧本与战场/剧本设计语义命令映射表.md`，输出 `设计语义 -> 剧本命令 -> 参数槽 -> 变量/防重复 -> 写回风险`；非 V1 安全命令必须标注完整树 / 备份 / 复读。
7. 写前复读：写包前复读人物、职业、物品、R/S、地图、Hexzmap、资源槽和变量用途；发现冲突时停止，不强行写入。
8. 编译包：优先走 `auto_make_mod(automation_mode=force_open)`；失败时转手动 `analyze_standalone_scenario_request -> compile_standalone_scenario_package -> preview_mod_package -> apply_mod_package(automation_mode=force_open)`。
9. 验证：当前项目必须验证 R 可打开、R 可跳 S、S 地图能载入、部署数量正确、胜败条件和战斗胜利测试存在、事件能触发、奖励不重复、阶段不会卡死。
10. 归档：导出报告时记录设计摘要、R/S 对照表、事件图、资源任务、写前/写后复读、验证结果、强制放行风险，作为下一关自动生成的知识样本。

## 六、AI 执行硬性规则

- “姜维传水准”是人物塑造质量标准，不是阵营或主角默认值。
- 用户只给一句主题时，可以自动补全人物、剧情、兵种、装备、特效、地图、资源风格和事件结构，但不得擅自指定固定主线。
- 6.5 前三关样本只作为结构参考；不得默认传统曹操传前三关，不得照搬样本文案、人物关系或数值。
- 生成单关 R/S 时，必须先输出战前情报卡、事件图、胜败条件变更表、坐标奖励表、待核验项，再写完整台词。
- R 剧本承诺的每个单挑、对话、奖励和特殊机制，都必须在 S 事件图中找到对应实现。
- 每个 S 事件必须同步输出“设计语义命令映射表”，按 `设计语义 -> 剧本命令 -> 参数槽 -> 验证方式` 查 `08-剧本与战场/剧本设计语义命令映射表.md`，不允许只写自然语言机制。
- S 剧本必须区分“玩家显示目标”和“实际判定事件”；条件变更必须写明触发节点和变更后失败条件。
- 自动化落地前必须查 `06-项目与工具链/自动化MOD制作.md`；独立基底低位编号、清表、资源门控、地图双层写入和 direct 写回规则以该页为准。
- 独立精简基底只能清空或覆盖定长表行，不能物理删除 `Data.e5 / Star.e5 / Imsg.e5 / Ekd5.exe` 表行；低位 Data 重占用只能用于复制出的独立基底。
- 自定义个人特技和兵种特技必须说明显示位置与实现路径：由 S 事件实现、由已验证装备特效实现，或列为待验证；不得伪造不存在的引擎特效。
- 地图必须区分底图和 Hexzmap 地形，`replace_map_image` 只替换视觉底图，`write_hexzmap_block` 才改变通行与地形判定。
- 没有通过资源预览和复读的头像、R 形象、S 形象、图标，不得写入最终资源编号；首版可以让形象字段保持 `0`。
- 独立单关默认走 `auto_make_mod(automation_mode=force_open)`；需要手动拆解时才走 `analyze_standalone_scenario_request -> compile_standalone_scenario_package`。
- `analyze_mod_request -> compile_mod_package` 只用于普通表/资源/特效包，不作为独立单关主路径。
- R/S 主结构必须由 standalone 编译器生成，不让自然语言模型直接手填二进制偏移。
- preview 失败优先自动修复并重试，不直接结束。
- direct 写回后必须输出备份、复读和报告路径；未通过验证的风险必须留在报告中，不得宣称已可发布。
- 任何强制放行都必须写进报告，不能在最终回复里隐瞒风险。
