# 特效号定位与安全读写：验证流程

## 证据顺序

1. 确认目标文件和当前 SHA256。
2. 按 PE 节表完成 VA、RVA 与文件偏移往返。
3. 完整解码指令，确认操作数真实偏移和宽度。
4. 恢复参数定义链、包装链、调用者和消费者。
5. 检查当前字节、共享调用者、代码洞所有权和冲突组。
6. 生成带 SHA 锁与每个物理位置旧字节锁的预览包。
7. 由当前进程签发短期一次性写入凭据，锁定项目身份、完整包哈希、绑定、元数据、写入段和旧字节；Apply 前验证正式发布组件一致性。
8. 在测试副本写入，写后复读字节并重新反汇编。
9. 重新扫描逻辑特效，确认编号、参数和自然语言含义可往返恢复。

## 写入能力

- 可直接修改：精确位置和编码已确认。
- 需要适配桩：值已确认，但原指令宽度或来源不适合原地修改。
- 需要整体事务：涉及多个位置或多个文件。
- 已确认，只读：语义成立，但共享影响或恢复路径不明。
- 仅供诊断：定义链、来源或当前安装证据不足。

当前默认可写 SHA256 为 `F9A357585F4D77C273A8BADD3E4EF4062983CC81D8040C0CB74768FAB7C04F5F`。`84E3A1DC...` 的动态证据属于旧基底，只能作为跨版本旁证。

## MCP 顺序

1. `scan_effect_id_locations`
2. `read_effect_id_location`
3. `preview_effect_id_update`
4. 复核返回包后显式调用 `apply_effect_patch`

普通用户不需要输入地址；地址、定义链和机器码只在高级诊断与 MCP 中出现。

## 动态证据边界

- GameDebug 新证据协议为 `EffectEvidenceBundleV1`。会话和原始采集固定保存在 `GameRoot/CCZModStudio_Notes/EffectContractEvidence/<sha>/<contractId>/`。
- `create_effect_probe_session -> run_effect_probe_batch -> capture_effect_probe_scenario -> finalize_effect_evidence_bundle -> import_effect_evidence_bundle` 是唯一能提升执行契约的路径。
- HMAC 正确仍不够；Core 每次使用证据都重新校验项目 ID、游戏目录、加载模块路径/大小/SHA、原始文件长度和摘要、场景身份、派生 JSON 值、能力 schema、构建通道和工具构建标识。
- 旧 `import_effect_contract_evidence` 与 `import_extended_binding_evidence` 只兼容保存诊断，不授予写入能力。
- 证据导入前先检查契约 hash、必需场景、槽位和合法边界；不合格证据不会写入 `.trusted.json`。
