> 用于曹操传 Mod 开发的安全跳转和代码空洞管理方案

---

#### 一、概述

本方案描述一种安全的代码修改方法：
1. 在原有代码位置使用 jmp 跳转到预找到的代码空洞
2. 在代码空洞中先恢复被覆盖的原指令
3. 然后添加新功能代码
4. 最后跳转回原地址继续执行
5. 保持原代码逻辑完整性

---

#### 二、已发现的代码空洞区域

##### 1. 空洞 A - 小型推荐 (推荐用于短代码)
- **地址区间**: `0x407675` - `0x4076FF`
- **大小**: 约 `0x8A` (138 字节)
- **填充内容**: 全部是 NOP (0x90)
- **位置**: 在正常代码流之间的安全区域
- **用途**: 适合放置中等长度的新代码

##### 2. 空洞 B - 小型
- **地址区间**: `0x407308` - `0x407357`
- **大小**: 约 `0x4F` (79 字节)
- **填充内容**: 全部是 NOP (0x90)
- **位置**: 在函数末尾的安全区域
- **用途**: 适合放置较短的新代码

##### 3. 空洞 C - 小型
- **地址区间**: `0x405B86` - `0x405B92`
- **大小**: 约 `0x0C` (12 字节)
- **填充内容**: 全部是 NOP (0x90)
- **位置**: 在函数之间
- **用途**: 非常短的代码，如简单的跳转中继

##### 4. 空洞 C2 - 微型 (🆕 已使用)
- **地址区间**: `0x43C2A0` - `0x43C2A9`
- **大小**: 约 `0x0A` (10 字节)
- **填充内容**: 全部是 NOP (0x90)
- **位置**: 在策略伤害计算函数内部的分支之间
- **用途**: "无视策略减伤"补丁的最短路径跳板

##### 4. 空洞 D - 大型 (⭐⭐⭐ 推荐用于复杂特技)
- **地址区间**: `0x4529A7` - `0x452F05`
- **大小**: 约 `0x55E` (1374 字节) 🔥🔥🔥
- **填充内容**: 全部是 NOP (0x90)
- **位置**: 在代码段中间，函数之间的大型预留区域
- **边界**: 
  - 起始: `0x4529A7` (紧跟在函数 return 之后)
  - 结束: `0x452F05` (0x452F06 是下一个函数的 push ebp)
- **用途**: 适合放置复杂特技的完整实现代码，可以容纳数十个函数调用
- **使用情况**: 部分已占用 (`0x4528FC` ~ `0x4529A6` 被"回MP攻击"补丁使用)

##### 5. 空洞 E - 超大型 (⭐⭐⭐⭐⭐ 主推荐用于大型扩展)
- **地址区间**: `0x601A9A` - `0x602500`+ (需要进一步确认结束位置)
- **大小**: 已确认至少 `0x866` (2150+ 字节) 🔥🔥🔥🔥
- **填充内容**: 全部是 NOP (0x90)
- **位置**: 在代码段后段，大型连续预留空间
- **扫描情况**: 从 `0x601A9A` 扫描到 `0x602500` 全部为 NOP
- **用途**: 适合放置需要大量代码的复杂系统级扩展

#### 三、安全跳转实现方案

##### 1. 标准跳转模板

假设我们要修改原地址 `0xXXXXXX` 处的代码：

```asm
; -----------------------------
; 原地址处的修改
; -----------------------------
0xXXXXXX:
  jmp 0x407675        ; 跳转到代码空洞 A
  ; [ 这里填充 NOP，确保不破坏原对齐 ]

; -----------------------------
; 代码空洞处的新代码
; -----------------------------
0x407675:
  ; 第一步：恢复被 jmp 覆盖的原指令
  ; (原指令需要先反汇编出来，在此处重写)
  
  ; 第二步：执行新功能代码
  ; [ 你的特技代码 ]
  
  ; 第三步：跳转回原地址
  jmp 0xXXXXXX + (被覆盖的指令长度)
```

---

##### 2. 实际例子（假设修改二次行动函数）

**修改前** (原地址 0x412D7B - 0x412D7F 已经是 NOP):
```asm
0x412D7B: 90 90 90 90 90   ; NOP 预留区域
0x412D80: E8 9B E3 FE FF   ; call 0x401120
```

**修改方案**:
```asm
; 0x412D7B 处跳转到代码空洞
0x412D7B: E9 xx xx xx xx   ; jmp 0x407675

; 在代码空洞 0x407675 处
0x407675:
  ; 1. 恢复原指令（0x412D80 处的 call）
  call 0x401120
  
  ; 2. 添加新特技代码
  ; [ 你的新代码 ]
  
  ; 3. 跳回 0x412D85 继续原流程
  jmp 0x412D85
```

#### 四、操作步骤

##### 步骤 1: 选择目标修改点
- 找到想要修改的程序逻辑
- 确定修改点的地址和原指令长度
- 确保原指令至少有 5 字节空间（可以容纳一个 jmp）

##### 步骤 2: 选择代码空洞
- 优先使用空洞 A (0x407675)，空间最大
- 如需多个补丁，可以分配不同的空洞区域
- 记录使用的地址范围，防止重叠

##### 步骤 3: 反汇编原指令
- 读取原地址处的代码
- 确保完全理解原指令含义
- 记录被覆盖的原指令，以便在空洞中恢复

##### 步骤 4: 写入补丁
- 原地址处写 jmp 到代码空洞
- 代码空洞处依次执行：
  1. 恢复原指令
  2. 写新代码
  3. 写 jmp 回原地址 + 偏移

##### 步骤 5: 测试验证
- 运行程序检查是否崩溃
- 验证原功能是否正常
- 验证新特技是否生效

#### 五、高级特性

##### 1. 多补丁管理
可以将代码空洞分段使用，例如：
- `0x407675` - `0x407695`: 特技 1
- `0x407696` - `0x4076B6`: 特技 2
- `0x4076B7` - `0x4076FF`: 特技 3

##### 2. 共享代码
如果多个特技有共用代码，可以在空洞中实现一次，其他地方调用

##### 3. 内存保护
- 只修改代码段，不碰数据段
- 使用 NOP 填充确保对齐
- 避免修改只读区域

#### 七、示例：在二次行动中添加新特技

> **注意**：以下仅为代码空洞跳转方案的演示示例。`0x412D7B-0x412D7F` 是 5 个 NOP，位于二次行动判定函数 `0x412D52` 内部，仅因该处存在可用的 NOP 空间而被选为演示点。**这绝不意味着二次行动函数可被其他特技复用或修改**；引入新特技应选择与原特技语义相关的 Hook 点。

##### 目标
在二次行动触发时，同时触发一个新的特技效果。

##### 修改方案
```asm
; 原地址 0x412D7B (已经是 NOP)
0x412D7B:
  E9 F5 48 FF FF      ; jmp 0x407675

; 代码空洞 0x407675
0x407675:
  ; 1. 先执行原本要执行的 call
  call 0x401120
  
  ; 2. 新特技代码
  ; [ 例如：检查某个条件，修改某些属性 ]
  
  ; 3. 跳回原地址 0x412D85
  E9 FB B6 00 00      ; jmp 0x412D85
```

---

#### 八、注意事项

1. **指令长度**: x86 指令是变长的，务必准确计算
2. **相对跳转**: jmp 使用相对地址，需要正确计算偏移
3. **寄存器保存**: 新代码如果修改寄存器，需要 push/pop 保存和恢复
4. **堆栈平衡**: 确保调用前后堆栈一致
5. **测试充分**: 每一个补丁都需要充分测试，避免引入 bug

#### 九、注入点选择原则 (🆕)

基于已注入补丁的实践经验，总结注入点选择的优先级：

1. **已有 NOP 区域**：最安全，无需处理被覆盖指令（如 `0x412D7B-0x412D7F`、`0x43C2A0-0x43C2A9`）
2. **函数尾部 NOP**：安全跳转，不会破坏任何逻辑
3. **非关键分支后的空闲空间**：确保有足够 5 字节放置 jmp
4. **需要恢复原指令的注入**：在代码洞中先执行被覆盖的指令（push/pop 栈上值），再执行新逻辑

##### 注入点选择检查清单

```
□ 目标地址 + 0x05 之后的代码是否仍在使用？
  → 是：需要在代码洞中恢复被覆盖指令
  → 否：直接 jmp 即可（如跳转到 NOP 区域后无需回跳）

□ 被覆盖的指令是否依赖特定的栈/寄存器状态？
  → 是：需要在代码洞中先恢复该状态

□ jmp 的目标代码洞是否在 ±2GB 范围内？
  → x86 相对跳转范围约 ±2GB，通常足够
```

#### 九、特技代码分配规则

##### 1. 独立区域分配原则

###### 规则 1：每个特技独占一个区域

##### 4. 特技代码写作提示 (🆕)

###### 寄存器使用规范

**规则：禁止修改 `ebx`、`esi`、`edi`、`ebp`（寄存器本身）。但通过 `[ebp-N]` 访问栈局部变量是安全且推荐的。**

| 寄存器/访问方式 | 类别 | 注入代码可用？ | 说明 |
|----------------|------|---------------|------|
| `eax` | caller-save | ✅ 随意使用 | 返回值/临时计算 |
| `ecx` | caller-save (thiscall) | ✅ 可用（注意 this 指针） | 调用 `0x4101D9` 等引擎函数前必须正确设置 |
| `edx` | caller-save | ✅ 随意使用 | 临时计算 |
| `ebx` | **callee-save** | ❌ 禁止使用寄存器 | 原调用者可能依赖其值 |
| `esi` | **callee-save** | ❌ 禁止使用寄存器 | 同上 |
| `edi` | **callee-save** | ❌ 禁止使用寄存器 | 同上 |
| `ebp` | 帧指针 | ❌ 禁止修改 `ebp` 本身 | 破坏栈帧指向将导致崩溃 |
| `[ebp-N]` | 栈局部变量 | ✅ 自由读写 | 用于临时存储/循环计数 |
| `[ebp+N]` | 函数参数 | ✅ 只读访问 | 读取原函数传入的参数 |

###### 替代方案

需要循环变量或临时存储时，使用栈局部变量 `[ebp-N]`，这在原版引擎中有大量先例：

- **`0x472D48`（属性下降函数）**：使用 `[ebp-0x04]` 做属性下降循环计数（`inc dword [ebp-0x04]`）
- **`0x43DADA`（物理攻击主函数）**：使用 `[ebp-0x0C]` 保存 this 指针，`[ebp-0x04]` 保存初始化结果
- **`0x412D52`（二次行动判定）**：使用 `[ebp-0x04]` 保存单位指针，`[ebp-0x08]` 保存状态标志地址
- **`0x438E78`（UI更新）**：使用 `[ebp-0x04]` 做循环计数器（`inc dword ptr [ebp-0x04]`）

```asm
; ✅ 正确：用 [ebp-0x04] 做循环计数（与原版 0x472D48 的 inc [ebp-0x04] 风格一致）
mov dword [ebp-0x04], 0
.loop:
  ...  ; call 0x409821(ecx, [ebp-0x04], 0x00)
  inc dword [ebp-0x04]
  cmp dword [ebp-0x04], 6
  jl .loop

; ❌ 错误：用 ebx 做循环计数（必须 push/pop，增加复杂度且不必要）
xor ebx, ebx
.loop:
  ...
  inc ebx
  cmp ebx, 6
  jl .loop
```

###### 原因

1. **x86 调用约定**：`ebx`/`esi`/`edi` 是 callee-save，调用者有合理预期这些寄存器在函数返回后不变
2. **代码洞继承调用者的栈帧**：代码洞实际上运行在原函数的栈帧中，必须遵守原函数的寄存器约定
3. **安全优先**：push/pop 可以解决，但增加了代码大小和出错概率。直接用 `[ebp-N]` 更简洁安全
4. **原版编码先例**：引擎本身大量使用 `[ebp-N]` 栈局部变量（详见替代方案中的原版函数列表），遵循同一风格降低维护成本
