# 6.4/6.5 运行时静态数据表地址

## 结论

用户补充的 `6.4一些地址.xlsx` 给出了道具、详细兵种、兵种系地形、消耗品数量和战场单位的连续地址样本。工作簿中的逐行地址不能全部直接采用，但以首项、相邻项差值和末项边界复算后，可恢复四张稳定的连续表。当前规范 6.5 `Ekd5.exe` 中的五个访问函数又独立复核了相同的基址和步长，因此这些表已进入 6.5 静态语义注册表；字段默认只读，不能仅凭地址表开放运行时写入。

规范 6.5 样本：

```text
文件：基底\加强版6.5未加密版\Ekd5.exe
SHA256：F9A357585F4D77C273A8BADD3E4EF4062983CC81D8040C0CB74768FAB7C04F5F
长度：1196032
```

## 来源审计

```text
原件：90-来源归档\原始文档\6.4一些地址.xlsx
SHA256：EF4F7704375B28081118C48C53BD64A76A2468F045EFE8C92873FEF04BC76F61
大小：37055 字节
```

| 工作表 | 有效范围 | 说明 |
|---|---:|---|
| 道具的地址 | 259 行，13 列 | 含道具 ID、名称和运行地址样本 |
| 兵种 | 81 行，20 列 | 含 120 个公式；N 列 79 处地址公式明显错误 |
| 消耗品数量地址 | 3 行 | 给出 ID 150 和 254 的边界样本 |
| 6.4战场内存地址 | 22 行 | 与 6.4 战术单位 `30H` 布局互相印证 |

工作簿同时混用十六进制文本、十进制数值和 Excel 公式。任何地址必须先统一成整数，再由“基址 + 记录索引 * 步长 + 字段偏移”复算。禁止把 `19H/23H/3CH/61H` 当成十进制 19/23/3/61。

## 连续表公式

| 表 | 地址公式 | ID 范围 | 记录宽度 | 6.5 访问函数 | 返回类型 |
|---|---|---:|---:|---|---|
| 道具记录 | `004A1140 + itemId * 19H` | `0..254` | `19H` / 25 字节 | `004837C2` | `ItemRecord*` |
| 详细兵种记录 | `004A2A27 + jobId * 23H` | `0..79` | `23H` / 35 字节 | `0041B782` | `DetailedJobRecord*` |
| 兵种系地形记录 | `004A3517 + familyId * 3CH` | `0..39` | `3CH` / 60 字节 | `0041B7D4` | `JobFamilyTerrainRecord*` |
| 策略记录 | `004A3E77 + strategyId * 61H` | 按当前策略目录 | `61H` / 97 字节 | `00484002` | `StrategyRecord*` |
| 消耗品数量 | `00510C80 + (itemId - 150)` | `150..254` | 1 字节 | `00414F65` | `ByteCount` |

### 边界复算

```text
道具 254：004A1140 + FEH * 19H = 004A2A0E
道具表尾：004A2A0E + 18H = 004A2A26

详细兵种 79：004A2A27 + 4FH * 23H = 004A34F4
详细兵种表尾：004A34F4 + 22H = 004A3516

兵种系 39：004A3517 + 27H * 3CH = 004A3E3B
兵种系表尾：004A3E3B + 3BH = 004A3E76
下一表：004A3E77（策略表）

消耗品 150：00510C80
消耗品 254：00510C80 + (254 - 150) = 00510CE8
```

这些边界连续且互不重叠。尤其 `004A3517..004A3E76` 恰好在 `004A3E77` 策略表前结束，是兵种系 `3CH` 步长的重要交叉证据。

## 字段布局

### 道具 `19H`

| 偏移 | 宽度 | 语义 |
|---:|---:|---|
| `+00` | `11H` | 名称存储区；文本容量 16 字节，另含终止/存储位 |
| `+11` | 1 | 类型 |
| `+12` | 1 | 装备特效号 |
| `+13` | 1 | 价格除以 100 |
| `+14` | 1 | 图标 |
| `+15` | 1 | 初始能力 |
| `+16` | 1 | 装备特效值 |
| `+17` | 1 | 成长值 |
| `+18` | 1 | 宝物图鉴标志 |

布局校验式：`17 + 8 * 1 = 25 = 19H`。旧口径若把名称当 16 字节并把类型放在 `+10`，会使后续字段整体错一字节。

### 详细兵种 `23H`

| 偏移 | 宽度 | 语义 |
|---:|---:|---|
| `+00` | 1 | 移动力 |
| `+01` | 1 | 攻击范围 |
| `+02..+06` | 5 | 攻、防、精、爆、士成长 |
| `+07/+08` | 2 | HP/MP 成长 |
| `+09..+22` | `1AH` | 26 类装备可用性矩阵 |

共 `9 + 26 = 35 = 23H` 字节，ID `0..79` 共 80 条。Excel“兵种”工作表 N 列的 79 处公式没有保持该步长，不能作为地址证据；应以本公式和当前 `0041B782` 为准。

### 兵种系地形 `3CH`

| 偏移 | 宽度 | 语义 |
|---:|---:|---|
| `+00..+1D` | `1EH` | 30 种地形发挥/适应值 |
| `+1E..+3B` | `1EH` | 同 30 种地形的移动消耗 |

HexTable 中“地形发挥”和“移动消耗”是同一 60 字节记录的前、后两个 30 字节视图，不是两张各自按 30 字节步进的独立记录表。

### 消耗品数量

数量区只覆盖物品 ID `150..254`。索引必须减 150；对物品 0 直接使用 `00510C80 + id` 会错读其他全局内存。数量为 BYTE，资料和当前访问函数尚不足以开放任意运行时写入。

## 战场单位补充

Excel 的“6.4战场内存地址”支持 `004A7B20 + battlefieldId * 30H`。共享 6.5 语义采用：

| 偏移 | 宽度 | 规范字段 |
|---:|---:|---|
| `+00` | WORD | `data_id`；`FFFF` 为空 |
| `+00` | DWORD | `data_id_container`，仅用于保留原始载体，不代表 Data号宽度 |
| `+04` | BYTE | `battle_sprite_id`，战场形象号 |
| `+04` | DWORD | `packed_display_state` 原始打包值 |
| `+05` | BYTE | 阵营 |
| `+06/+07` | BYTE | X/Y |
| `+10/+14` | DWORD | 当前 HP/MP |

例如 `+04..+07 = 34 02 0A 0B` 时，`battle_sprite_id=34H`，`side=02H`，`x=0AH`，`y=0BH`，`packed_display_state=0B0A0234H`。不得把整个 DWORD 命名为独立“显示编号”。

## 工具落地

- `EngineRuntimeSemanticRegistry` 是 6.5 唯一运行时语义源，登记上述表、字段、函数和单位解码宽度。
- 主程序的 `RuntimeTableSemanticService` 可按表 ID、记录 ID、字段 ID 解释地址。
- GameDebug 的 `game_read_runtime_table` 只读工具支持 `items`、`detailed-jobs`、`job-family-terrain`、`consumable-counts`。
- `RuntimeTableSchemaAuditService` 校验 HexTable 的道具 25 字节、详细兵种 80x35 和地形 2x30 视图。
- 6.4 战场快照只作为外部资料支持的只读档案；自动化、证据提升和写入仍要求已验证 6.5 布局。

## 安全边界

- 当前地址是规范 6.5 映像虚拟地址，不是文件偏移。
- 表字段进入共享注册表不等于可写；登记字段当前默认 `Writable=false`。
- `00484002` 只能返回 `StrategyRecord*`，不得满足 `RuntimeCharacter*` 或 `TacticalUnit*` 写入条件。
- Excel N 列错误、十六进制/十进制混填和公式缓存值只能作为资料质量警报，不能由工具自动纠正后写回原件。
- 正式 EXE 写入继续受 SHA、旧字节锁、结构化契约、事务和 V3 证据共同约束。
