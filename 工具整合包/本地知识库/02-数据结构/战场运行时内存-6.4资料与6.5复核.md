# 战场运行时内存：6.4 资料与 6.5 复核

## 结论速览

- 用户提供的《★ 6.4 战场内存记录》与 godtype 的 OllyDbg 教程补齐了战术单位、物理攻击、策略、道具、人物运行时记录和战场全局变量之间的关系。
- 本轮最重要的纠错是：`004A3E77 + 策略号 * 61H` 是**策略记录表**，`00484002` 是**策略号转策略记录指针**；它不是人物 Data 数组，也不是 Data 号转人物指针。
- 当前 6.5 的人物运行时记录由指针槽 `[004CEA00]` 给出基址，每人 `48H` 字节；`004061E4` 按 Data 号取人物记录，`0040658F` 可从战术单位指针转到人物运行时记录。
- 当前 6.5 的战术单位数组仍为 `004A7B20`、每槽 `30H` 字节。`unit+00` 的低 2 字节是 Data 号，`FFFF` 表示空槽；`unit+04` 是出场/显示编号载体，不得再标为 Data 号。
- 这些资料解决了“取错结构、把策略号当 Data 号、把当前 MP 当最大 HP”的知识缺口，也显著改善特效代码中的上下文取值设计；但它们没有自动补齐 Hook 触发语义、覆盖旧字节、寄存器/标志保护、代码洞占用和 V3 动态证据，因而不能单独把任意新行为特效升级为可写。
- “不改工具源代码”时，现有 v7 工具链可继续处理已登记数据、直接参数和已有适配器；“完全不改 `Ekd5.exe` 代码字节”时不能称为特效注入，只能做已有表项/参数配置。

## 适用版本

- 主来源文档标注为 6.4，地址和字段先按 **6.4 外部资料**处理。
- 本页列为“当前 6.5 已确认”的项目，必须同时有当前规范 EXE 静态反汇编、本地旧工具源码或 6.5 x32dbg 动态证据。
- 论坛教程最初发表于 2009 年，正文展示的是原版/早期扩展引擎。其方法论和结构语义可以参考，`004B2C50/24H`、`004A1B68`、`004AF4C0` 等旧地址不能套用当前 6.5。
- 当前 6.5 规范 EXE：SHA256 `F9A357585F4D77C273A8BADD3E4EF4062983CC81D8040C0CB74768FAB7C04F5F`，档案 `ccz65-unencrypted-effect-authoring-v3`，只读审计结论为 `ExactCanonical`。

### 关键静态纠错证据

当前 6.5 `00484002`：

```asm
00484002  push ebp
00484005  imul ecx, ecx, 61
00484008  add  ecx, 004A3E77
0048400F  ret
```

`0044A5EB` 先从 `[00497AF8]` 读取当前策略号，再调用 `00484002`；这与 6.4 文档的“策略内存、每策略 `61H` 字节”一致，推翻了旧知识库的“Data号转角色数据”命名。

当前 6.5 `004061E4`：

```asm
004061E7  movzx ecx, word ptr [ebp+08]
004061EB  imul  ecx, ecx, 48
004061EE  add   ecx, dword ptr [004CEA00]
004061F5  ret   04
```

当前 6.5 `0041DF70`：

```asm
0041DF73  push dword ptr [ecx] ; unit+00 的 Data 号载体
0041DF75  call 004061E4       ; Data号 -> 人物运行时记录
0041DF7A  call 00406610       ; 读取人物记录 +2B 的兵种
```

这三段代码共同区分了“策略记录 `61H`”和“人物运行时记录 `48H`”两套结构。

## 实现/使用方法

### 地址必须先分类

| 分类 | 示例 | 使用规则 |
|------|------|----------|
| 运行时静态地址 | `004927F0`、`004A7B20` | 仅对匹配版本/映像基址的进程有效，不等于文件偏移 |
| 运行时指针槽 | `004CEA00` | 先读槽中的指针，再按步长定位；不能把槽地址当数组基址 |
| 运行时动态堆地址 | 文档中的“`2F1000` 附近浮动” | 每次运行可能变化，只能由指针链或函数取得 |
| EXE 代码地址 | `004061E4`、`00484002` | 作为函数/Hook 候选时还需记录原字节、ABI 和当前 SHA |
| 旧版固定地址 | `004B2C50`、`004A1B68` | 只用于识别结构演化，不写入 6.5 默认路径 |

### 当前 6.5 战术单位记录

计算式：

```text
unit = 004A7B20 + battlefield_id * 30H
```

| 偏移 | 推荐语义 | 当前证据与边界 |
|------|----------|----------------|
| `+00..+01` | Data 号；`FFFF` 为空槽 | 当前 EXE `cmp word ptr [unit],FFFF`、`004061E4` 与旧扳手源码共同支持 |
| `+02..+03` | Data 号载体高位/扩展字段 | 多处按 DWORD 读取，但高 2 字节独立语义未确认 |
| `+04` | 出场/显示编号载体 | 6.4 文档与旧扳手一致；当前 `00412D9D` 把 DWORD `[unit+04]` 传给 UI 更新函数；不是 Data 号 |
| `+05` | 阵营：`00` 我军、`01` 友军、`>=02` 敌军 | 旧工具源码 + 当前战场读取实践 |
| `+06/+07` | 当前 X/Y 坐标 | 当前场景读取支持；写入函数仍需“只移动”样本闭环 |
| `+08` | AI 攻击/跟随目标战场编号；`FF` 表示用坐标目标 | 6.4 文档 + 旧工具源码，6.5 待分 AI 样本验证 |
| `+09/+0A` | AI 移动目标 X/Y | 同上 |
| `+0B` | AI 行动顺序 | 外部佐证，6.5 待动态 |
| `+0C` | 显示/退场状态：`01` 隐藏、`02` 可见、`03` 退场；`04` 可见变体待证 | 6.4 文档 + 旧工具源码 |
| `+0D` | 行动状态位；旧工具以 `| 06` 标已行动、`& F8` 清行动位；`08` 与 S 形象翻转相关 | 当前行动逻辑 + 旧工具源码；每一 bit 仍需分动作验证 |
| `+0E` | AI/控制策略 | `0/1/2/3/4/7` 的完整含义见下表；不是单纯布尔值 |
| `+0F` | 朝向：`0` 上、`1` 右、`2` 下、`3` 左 | 6.4 文档 + 旧工具源码 |
| `+10..+13` | 当前 HP，DWORD | 当前 6.5 `0043F70C` 最终写 `dword ptr [unit+10]`，已动态闭环 |
| `+14..+17` | 当前 MP，DWORD | 6.4 文档 + 旧工具源码 + 6.5 补丁样本；MP setter 尚未动态闭环 |
| `+18..+1D` | 攻、防、精、爆、士、移动的临时档位 | 旧工具与补丁样本一致；具体档位衰减需动态 |
| `+1E` | 异常状态组合 | 6.4 文档、旧工具与注入样本一致 |
| `+1F` | 是否已说撤退台词 | 6.4 文档外部佐证；旧知识库曾误作 MP 字段，6.5 待动态 |
| `+20/+21` | 限制区域或第二目标坐标 | 两个来源口径冲突，禁止直接写 |
| `+22` | 未知/扩展字段 | 旧工具只显示原值 |
| `+23` | 戮力同心次数或回合扩展状态 | 两个来源口径未闭环，当前只确认新回合路径会清该字节 |
| `+24..+2F` | 6.4/6.5 扩展区 | 未建立字段级写入规则 |

### 对特效注入的直接用法

新行为草案应按以下顺序取上下文：

1. 先判断当前 Hook 属于物理、策略还是道具路径。
2. 从对应上下文读取战场编号或已经缓存的战术单位指针。
3. 需要 HP、MP、坐标、阵营和状态时使用战术单位记录。
4. 需要兵种、装备、基础五维和人物持久属性时，用 `unit+00` 的 Data 号经 `004061E4/0040658F` 取得人物运行时记录。
5. 需要策略属性时，用策略号经 `00484002` 取得 `61H` 策略记录，绝不能把它当人物记录。
6. 再确定 Hook 覆盖、原指令迁移、返回地址、寄存器和 EFLAGS 保护；内存字段正确并不等于 Hook 契约完整。

## 风险与边界

- 本页已纠正结构身份，但没有修改 CCZModStudio/GameDebug 源码中仍沿用旧命名的 `get_char_data_ptr`、`UnitDataIdOffset=0x04` 等实现。代码在修正和重新测试前，相关自动解释不能升级为可信写入证据。
- Word 文档内有问号、段落错位和明显旧口径。例如 `004A3E77` 的标签正确，但若把 `61H` 当十进制 61 会再次出错；这里的 `61H` 是十六进制 97 字节。
- 论坛教程的 `004B2C50 + id * 24H` 是早期结构；当前 6.5 已扩成 `004A7B20 + id * 30H`。旧函数 `00407A09/00407AF2` 和旧代码洞也不能直接套用。
- 内存直接写可以绕过游戏刷新、范围检查、撤退/死亡逻辑和派生属性重算。即使字段已确认，也应优先调用已验证的内部 setter 或使用剧本/表配置。
- “注入”必然修改 `Ekd5.exe` 代码字节或运行时代码。若要求游戏 EXE 也完全不变，只能使用原生表项、已有特技参数、人物/兵种/宝物绑定和剧本变量，不能增加新的行为分支。
