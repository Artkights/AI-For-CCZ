# 13.1 旧剧本编辑器源码对照

## 2026-06-02 已落地的旧版完整树实现

- `LegacyScenarioReader` 已按旧版 C++ 规则读取 R/S eex：
  - 固定 EEX 头和 Scene 偏移表。
  - Scene 内 2 字节 Section 数。
  - Section 2 字节长度前缀。
  - 命令参数按 `code_instruct` 读取。
  - `0x46` 使用 11 * 20 参数循环，`0x47` 使用 12 * 80 参数循环。
  - `0x01 + code_test[id] != 0` 生成子事件载体，读取其后 2 字节子块长度并递归解析。
  - `0x76` 读取后按 `pos + dif + 8` 反查目标命令 ord。
- `LegacyScenarioWriter` 已按旧版保存语义重建 R/S eex：
  - 写 10 字节头并回填 Scene 偏移表。
  - 回填 Section 和子块 16 位长度。
  - 按递归命令树写命令和参数。
  - 保存末尾按新命令偏移回填 `0x76` 相对位移。
  - 写入正式项目前自动备份，临时文件替换前先按旧版 Reader 重读校验。
- 剧本制作页已优先绑定 `LegacyScenarioDocument`：
  - 旧版读取成功时，左树来自真实 Scene/Section/子块结构。
  - 可视树按旧版 `CreateFileTree()` 风格只显示“文件路径 -> Scene -> Section -> 真实指令节点”；`正文根`、`测试条件`、`事件正文`、`子事件主体/载体`、独立 `文本参数` 只作为内部语义和写回模型保留，不再作为额外中文包装节点显示。
  - 文本参数直接体现在所属指令行和旧式 Dialog 中；右侧备注区只写项目侧创作者备注，不写入 `.eex`。
  - 旧版读取失败时才回落到 `ScenarioStructureProbeReader` 兼容探针。
- 当前烟测：
  - `--rs-smoke` 输出 `LEGACY_SCENARIO_READ`。
  - `--rs-write-smoke` 输出 `LEGACY_SCENARIO_WRITE_OK`。
  - 当前 `R_00.eex` 未修改完整写回为 `changedBytes=0`。

本文只记录已经从 C++ 源码直接确认的结构事实，优先级高于之前基于探针和 UI 现象的推测。

## 1. eex 文件头与 Scene 索引

`cczEditor2Doc.cpp::OnSaveDocument`

- 保存时固定先写入 10 字节头：`45 45 58 00 01 02 00 00 00 00`。
- 紧接着为每个 Scene 预留 4 字节偏移表。
- 每个 Scene 实际起点会回填到 `10 + sceneIndex * 4`。
- 每个 Scene 内部先写 2 字节 `section_cnt`，然后顺序写每个 Section。

结论：

- 当前工具把 `.eex` 视为 `EEX\\0 + Scene offset table + Scene body` 是对的。
- `HeaderSize` 不能再只当“候选”；旧工具的保存逻辑明确以 `10 + sceneCount * 4` 形成头部。

## 2. Section 是带长度前缀的嵌套块

`cczEditor2Doc.cpp::OnSaveDocument`

- Section 开头先写 2 字节总长度占位。
- Section 内命令顺序写入。
- 一旦某条命令带子节点，就继续进入子节点并额外写一个 2 字节长度占位。
- 当子块结束时，回填 `当前块长度 = endPos - blockStart - 2`。

结论：

- Section 本身不是简单命令数组，而是“长度前缀块 + 可递归子块”。
- 旧工具里的子事件嵌套，底层依赖的是“命令后可接一个子块长度+子命令流”。
- 当前工具仍不能把探针结果直接当可写命令树。

## 3. 旧工具读取时的事件/子事件建树规则

`cczEditor2View.cpp::CreateFileTree`

已确认：

- `head=true` 时处于 Section 头部。
- 遇到 `id == 0` 且 `head == true`：
  - 头部结束；
  - 当前父节点切到这个 `0x00` 命令之下；
  - 额外跳过 2 字节长度。
- 遇到 `id == 1`：
  - 仅设置 `zsj_flag = true`，表示“刚看到子事件设定”。
- 下一条命令若 `code_test[id] != 0` 且 `zsj_flag == true`：
  - `zsj_sum++`；
  - 当前父节点切到“这条命令”之下；
  - 额外跳过 2 字节长度。
- 遇到 `id == 0` 且 `zsj_sum > 0`：
  - 当前子事件块结束；
  - 父节点回到上一层。

关键结论：

- `0x01` 本身只是“子事件设定”标志，不是直接承载子树的节点。
- 真正挂子树的是 `0x01` 后面那条满足 `code_test[id] != 0` 的命令。
- `0x00` 有两种角色：
  - Section 头部结束后，作为“正文根”并带一个子块长度；
  - 子事件块结束标记。

因此，当前工具的数据模型应保留下面语义，但可视 UI 不应额外制造这些中文包装节点：

- 旧源码实际只插入 `Scene`、`Section` 和每条真实指令节点。
- `0x00` 作为 Section 头部结束时，源码把后续父节点切到这个 `0x00` 指令下，而不是再创建“正文根/事件正文”节点。
- `0x01` 作为“子事件设定”时只设置标志；它后面满足 `code_test[id] != 0` 的命令才承接子块，源码直接把父节点切到该命令下，不创建“子事件主体/载体”包装节点。
- UI 对齐旧版时应显示为 `文件路径 -> Scene -> Section -> 指令`，子事件通过真实指令节点嵌套；内部语义写入 tooltip/详情或数据模型，不抢占树视图。

## 4. 无条件跳转 0x76 的真实保存语义

`cczEditor2Doc.cpp::OnSaveDocument`
`cczEditor2View.cpp::CreateFileTree`

读取时：

- `code_off[ord] = 命令实际文件偏移`
- `0x76` 先读出相对位移 `dif`
- 目标偏移按 `pos + dif + 8` 计算
- 再在 `code_off[]` 中反查对应命令 ord，写回 `int_data[0]`

保存时：

- 保存前先记录每条命令新 ord
- `0x76` 保存时先记下当前位置 `jmp_pos`
- 文件写完后回填 4 字节跳转位移：
  - `targetOffset - currentOffset - 8`
- 保存完成后再按新 ord 刷新树中显示

结论：

- 旧工具里的 `0x76` 不是简单“事件块编号”，而是基于命令偏移/ord 的跳转。
- 当前工具把它泛化成“跳转目标候选”可以保留，但要明确这是“命令序号/偏移回填语义”，不是普通文本编号。

## 5. `code_instruct` / `code_test` 的直接价值

`cczEditor2View.h`

- `code_instruct[124][13]`：旧工具的命令参数布局表。
- `code_test[124]`：旧工具对命令类别/可否作为子事件载体的判定表。

直接可用结论：

- 现有 C# `ScenarioCommandParameterTemplateService` 可以继续借用命令名和用途，但后续若要逼近旧工具，应该把 `code_instruct` 逐步转为结构化参数长度表。
- `code_test[id] != 0` 可作为“这条命令可承接子事件块”的已确认信号，不应再只靠 `0x01` 单独入栈。

### 2026-06-05 战场部署大块的特殊长度

- `code_instruct` 每条只有 13 个槽，但旧工具对部分命令不是简单读取一遍：
  - `0x46 友军出场设定`：按前 11 个布局槽循环 20 组，合计 220 个参数。
  - `0x47 敌军出场设定`：按前 12 个布局槽循环 80 组，合计 960 个参数。
  - `0x4B 我军出场设定`：按 5 个布局槽读取一条记录，不是大块循环。
- 任何只读探针或候选表都必须同步这两个循环规则。否则 `46/47` 会被当作短命令截断，后续命令边界整体错位，战场编辑器会表现为“能看到出场命令名，但人物、坐标、AI 解析不正确”。
- 旧结构参数不是全 16 位：`0x04` 布局槽是 32 位整数参数。`0x46` 的 X/Y 分别是第 3/4 个逻辑参数，`0x47` 的 X/Y 分别是第 4/5 个逻辑参数，均来自 32 位槽；如果只展开 16 位词或按参数对象下标错位写入，会把出场标志误当坐标。
- 当前 `LegacyScenarioReader` 与 `ScenarioStructureProbeReader` 均已同步该规则；战场页把 `46` 拆为最多 20 条友军候选、`47` 拆为最多 80 条敌军候选、`4B` 拆为单条我军候选。`4B` 按 docx 是我军出战选择排列，不是直接地图坐标。
- 字段写回边界：只对已有 `46/47` 记录开放受控写回，人物和 AI 写 16 位槽，X/Y 写 32 位槽；`4B`、纯拖放新增单位、等级/装备/隐藏/援军等未完全确认字段仍只做候选核对和项目侧备注。

### 2026-06-06 旧版 MFC Dialog 修改链路复刻

本轮以旧源码 `cczEditor2View.cpp::OnEditModify()`、各 `Dialog_xx::OnInitDialog/OnBnClickedOk`、`cczEditor2View.h` 的 DDX 映射和 `cczEditor2.rc`/`DLGINIT` 为准，整合包的正式单条指令修改链路改为：

- `eex 二进制 -> LegacyScenarioReader -> Scene/Section/CommandBlock -> LegacyScenarioItemData`
- TreeView 节点 `Tag` 绑定 `LegacyScenarioItemData`，修改入口不再以通用参数表为主体。
- 打开旧式编辑器时先复制 `LegacyScenarioItemData.CloneSnapshot()` 作为工作对象，OK 后才写回 `IntData` / `LongCharData`；Cancel 不改原对象。
- 剧本制作、战场制作 S 剧本、R 场景制作 R 剧本共用旧式树 Presenter 和同一套 Dialog 分派/保存链路；不同页面只负责选择当前 `LegacyScenarioDocument` 和对应完整保存按钮。
- `LegacyCommandEditDispatcher` 按旧 `OnEditModify()` 分派命令号到旧窗口名；旧源码覆盖 101 个命令号、71 个旧 Dialog，其中 `0/1/2/20/22..26/105/122` 均属于 `Dialog_2` 文本类窗口。
- `LegacyMfcDialogResourceLoader` 直接解析旧 `.rc` 的 `DIALOGEX`、控件文案、控件类型、坐标、`CBS_SORT`、`DLGINIT` 下拉项，并从旧 `Dialog_114::OnBnClickedButton1()` 抽取 34 个信息传送模板。
- `LegacyMfcDialogCatalog` 为 71 个旧窗口建立显式行为目录，复刻初始化、下拉/勾选/列表交互和 OK 写回槽位；窗口外观由统一 WinForms 渲染器承载，但控件文案、下拉项和业务写回规则来自旧资源/旧源码。
- 文本类写回按旧版：显示时内部 `\n` 转多行编辑框，OK 时删除 `\r` 并保留 `\n` 写回 `LongCharData`。
- `Dialog_6`、`Dialog_70` 的列表项高度按旧 owner-draw 口径固定为 18。
- `Dialog_78` 的 `IDC_COMBO9` 按旧源码 `combo5.GetCurSel()` 直接写 `int_data[8]`，不是 `Per2List2Code`。
- `Dialog_89` 初始化会把负等级槽 `int_data[2/4/6]` 归零；物品槽按旧规则 `p % 256 != 255 ? p : -1` 保存。

当前保守边界：

- 本轮只迁移“单棵树、单条指令修改”，大规模修改和结构编辑仍不作为旧式 Dialog 的能力目标。
- 通用参数表保留为参考/调试视图，不作为正式修改入口。
- 保存仍统一走 `LegacyScenarioWriter`，即按 `code_instruct` 模板整体重新编码，而不是保存旧参数行里的 tag。

## 6. 当前整合包的实施边界

旧版完整树主线已落地后，当前工具的稳妥路线是：

1. 列表页继续只扫 `RS/R_*.eex`、`RS/S_*.eex`。
2. 旧版完整树读取成功时，文本参数走完整结构保存；兼容探针路径仍只开放“原偏移、原容量内 GBK 短写回”。
3. 事件树 UI 保持贴近旧工具的可视层：`文件路径 -> Scene -> Section -> 指令`。条件区、正文根、子事件设定/载体是内部解析语义，不再作为额外树节点显示。
4. 说明和源码对照里明确：
   - `0x76` 基于命令偏移/ord 回填
   - `0x01` 只是子事件设定
   - `0x00` 在头部和子事件结束处语义不同
5. 旧版完整树模式下已开放保守普通命令编辑：
   - 可从 `CczString.ini` 选择普通命令，添加到正文末尾、在普通命令前后插入、追加到已有子块。
   - 可删除无子块、未被 `0x76` 引用、且非结构分界的普通命令。
   - 可编辑非结构/非跳转命令的普通数值参数：`Word16` 与非 `0x76` 的 `Dword32`。
   - 正文根、子事件设定/载体、`0x0C`、`0x0D`、子事件结束 `0x00`、`0x76` 跳转、可变数组和跨 Section 重排仍保持保护。
6. 所有节点增删和参数修改先写入内存中的 `LegacyScenarioDocument`，必须点击 `完整保存剧本` 才落盘；写入继续依赖备份、临时文件、旧版 Reader 复读校验和结构化报告。

### 2026-06-06 `UpdateShow()` 中文摘要迁移

旧源码 `cczEditor2View.cpp::UpdateShow(HTREEITEM me)` 不是简单把参数列表拼到树节点上，而是按命令 id 分支把 `int_data` / `long_char_data` 映射成创作者可读文本。本次迁移新增 `LegacyScenarioCommandDisplayFormatter`，并让剧本、战场、R 场景三处旧版脚本树共用。

关键口径：

- 文本类命令把内部换行显示为 `\n`，直接挂在命令行后，不再创建独立“文本参数”子节点。
- `0x05` 变量测试按旧版 25/25 两组变量数组显示，空组显示 `无`。
- 人物选择走旧版 `Per1CodeToList` / `Per2CodeToList`，显示人物名或特殊对象，不显示内部负数编码。
- 坐标、阵营、天气、物品、兵种、状态、比较符、运算符、单挑动作等均优先使用 `LegacyMfcDialogDataSources` 从旧版 `CczString.ini` 与项目资源加载的中文映射。
- 注意旧源码里的 `id == 119/120/121` 对应当前命令字节 `0x77/0x78/0x79`，不能误按十进制 `77/78/79` 迁移。
- `0x76` 在当前 C# 读写链路仍按无条件跳转 ord/偏移回填语义处理；显示层只负责摘要，不改变保存语义。

已覆盖验证样例：

- `0x05 变量测试`：显示 `Var20;无`。
- `0x25 武将进入指定地点测试`：显示中文人物/阵营与坐标 `(13,12)`，不出现 `P0=`。
- `0x36 武将状态测试`：显示人物名、`HPCur` 与比较符 `=`。
