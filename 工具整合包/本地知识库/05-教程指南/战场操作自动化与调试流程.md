# 战场操作自动化与调试流程

## 结论速览

- 当前最快的动态采证流程已升级为“人类正常游玩 + 只读自动监控器”：用户正常推进战斗，调试侧用 `watch-6.5-battle-events.ps1` 持续读取单位数组并自动记录事件。
- “人类操作员 + 调试采证员”的单动作协议仍适合精确复核；但日常探索优先开常驻监控，减少人工往返。
- 战场操作不得依赖盲目点击。每一步都应先确认单位、坐标、攻击范围、行动状态和目标是否可选。
- 2026-06-09 颍川之战实机确认：选中我军后，紫色格子是可移动目标，红色框格子是攻击范围；攻击前必须确认敌军位于我军攻击范围内。

## 适用版本

- 默认适用：曹操传加强版 6.5（未加密）当前项目基底。
- 本页描述的是当前 6.5 基底的实机操作与调试采证流程；自动化脚本或进程内桥接只能在这些规则被进一步验证后实现。

## 已确认事实

### 战场 UI 操作规则

- 点击我军单位后，单位周围的紫色格子表示该单位当前可以移动到的位置。
- 再次点击紫色格子，且目标格子没有单位占据时，单位会移动到该位置。
- 点击我军单位后，单位周围的红色框格子表示该单位当前攻击范围。
- 攻击必须满足目标敌军位于红色框攻击范围内。
- 除二次行动、再次移动等特技外，人物一回合通常只能执行一次“移动 + 行动”。
- 移动是改变自身位置，可不执行；行动是攻击敌军、释放策略、使用道具或待命。
- 移动必须发生在行动之前。完成行动后，该单位本回合不能再次操作。
- 已完成行动的我军单位形象会从高亮变为灰暗；点击该单位会触发提示：“已经行动完毕。”
- 执行物理攻击的人工流程：
  1. 点击我军单位。
  2. 若敌军已经在红色框内，不移动单位。
  3. 若敌军不在红色框内，先选择一个无人占据的紫色格子移动，使目标进入攻击范围。
  4. 点击右侧最上方“攻击”指令。
  5. 点击红色框范围内的敌军目标。
  6. 等待动画、反击、功勋/经验提示全部结束后停止操作。

### 调试采证规则

- 调试侧默认不主动暂停游戏，避免 x32dbg 令 `Ekd5.exe` 窗口无响应。
- 常驻自动监控优先使用只读 `ReadProcessMemory` 读取 `Ekd5.exe`，不设置断点、不暂停游戏、不写游戏文件。
- x32dbg MCP bridge 在线时，自动监控器只额外记录 bridge 状态和暂停点证据；断点仍需单独明确启用。
- 需要暂停、设置断点或写断点时，必须提前告知操作者，并在设置完后立即继续运行。
- 一次只采一个动作。动作完成后先回复固定短语，再由调试侧抓后快照。
- 快照优先范围：
  - `char_data_004A3E77`：角色数据前 160 条，大小 `0x3CA0`。
  - `unit_array_004A7B20`：战术单位数组。
  - `battle_globals_00490000`：战斗相关全局区。
  - 运行时 heap 只作辅助；读取失败或无差异时不得作为否定证据。

## 实现/使用方法

### 常驻自动监控协议

1. 用 `工具整合包\MCP配置\start-x32dbg-6.5.ps1` 打开正式 6.5 未加密基底；脚本默认按目标 EXE 哈希定位正式基底，并排除 `CCZModStudio_TestCopies`。
2. 启动监控：

   ```powershell
   powershell -NoProfile -ExecutionPolicy Bypass -File "工具整合包\MCP配置\watch-6.5-battle-events.ps1" -DurationSec 1800 -PollIntervalMs 500 -Label "merit-auto" -SaveMemoryOnEvent
   ```

3. 用户正常游玩，不必每一步停下来汇报。
4. 监控器自动输出到 `CCZModStudio_Reports\DebugEvidence\merit-auto-YYYYMMDD-HHMMSS`：
   - `events.md`：人工阅读流水。
   - `events.jsonl`：单位差分事件，适合后续脚本分析。
   - `summary.json`：轮询次数、事件数、错误数。
   - `event-0001/` 等：启用 `-SaveMemoryOnEvent` 时保存事件瞬间关键内存块。
5. 若需要函数调用栈，再围绕已发现的事件时间点开启有限断点或写断点复核。

### 人类操作员协议

1. 调试侧说明当前目标动作和停止条件。
2. 调试侧抓前快照。
3. 用户只执行该动作，不额外选中其它单位、不推进其它回合。
4. 用户回复动作完成。
5. 调试侧抓后快照并做 diff。
6. 调试侧将 UI 观察、内存差分、候选地址和下一步写断点计划写入证据。

### 战场状态读取口径

- 战术单位数组基址：`0x004A7B20`。
- 战术单位 stride：`0x30`。
- 当前已用字段：
  - `unit+0x04`：Data 号低字节候选。
  - `unit+0x05`：阵营候选，`0=我军`、`1=友军`、`2=敌军`。
  - `unit+0x06/+0x07`：战场坐标候选。
  - `unit+0x0D`：行动状态。
  - `unit+0x10`：当前 HP。
  - `unit+0x14`：当前 MP 候选。
- 战场单位身份必须以游戏画面、剧情上下文和用户实机确认为准；不能把 `unit+0x04` 的 Data 号低字节直接等同于当前画面人物名。
- 2026-06-09 颍川之战基线：
  - 曹操：`unit[0]`，坐标 `(10,6)`，HP `167`，MP `38`。
  - 张梁候选：`unit[61]`，坐标 `(10,5)`，HP `168`；曹操攻击后 HP 降为 `122`。
  - 用户纠错：当前关卡没有郭嘉、荀彧；此前按 Data 号低字节把 `unit[20]`、`unit[22]` 映射成人名属于不可靠推断。
  - 曹操在本关可释放策略，可作为策略使用与智力功勋样本的候选操作者；但若曹操已完成本回合行动，必须先由画面确认是否仍可操作。
  - 曹操完成攻击后，本回合不可再操作；点击曹操会提示“已经行动完毕”。

- `game_battle_state_match(profile="yingchuan_cao_zhangliang")` 已作为无鼠标、无截屏的只读 profile gate 接入 `CCZModStudio.GameDebugMcpServer`。
- 该工具只读 `unit_array=0x004A7B20`，检查 2026-06-09 颍川样本：曹操 `unit[0]`、阵营 `0`、坐标 `(10,6)`、HP 大于 0；张梁 `unit[61]`、阵营 `2`、坐标 `(10,5)`、HP 大于 0。

- 该门禁不发送输入、不截图、不写进程内存、不调用调试器命令；输出 `battle-state-match.json/md` 和 `events.jsonl`。
- 该门禁只能证明“当前运行态匹配颍川曹操/张梁样本”或“已观察到攻击后 HP 下降”。攻击前、攻击后、回合结束等函数语义仍必须由 x32dbg 动态命中、调用栈、寄存器、反汇编和战场状态差分共同闭环。
- `debug_battle_profile_probe_run(profile="yingchuan_cao_zhangliang")` 是当前推荐的 profile-gated 探针编排入口。安全模式默认 `run_probes=false`，只生成 `battle-profile-probe-run-summary.json/md`、地址报告和探针计划；实机调试时只有在 x32dbg bridge 在线且 profile 已匹配后，再显式传入 `run_probes=true` 运行攻击/回合阶段断点。
- 无游戏进程时，该工具应返回 `profile-not-ready-plan-ready`。这表示当前不能动态验证攻击/回合语义，但地址探针计划已经可供进入颍川样本后复用。
- `debug_live_probe_readiness(profile="yingchuan_cao_zhangliang")` 是运行动态 probe 前的标准预检。它只读检查目标进程/窗口、x32dbg bridge、运行态分类、颍川 profile 和探针计划，输出 `live-probe-readiness.json/md`。
- 只有 readiness 报告 `ready_for_dynamic_probe=true` 后，才应运行 `debug_battle_profile_probe_run(..., run_probes=true)` 并开始解释攻击前、攻击后、回合结束函数命中。无进程时 readiness 会返回 `not-running`，并列出缺失 gate，当前验证样本缺失 `process_running`、`window_detected`、`x32dbg_bridge_online`、`runtime_battle_loaded`、`battle_profile_ready`。
- `debug_live_probe_auto_run(profile="yingchuan_cao_zhangliang")` 是当前一步式实机入口。安全模式默认只记录 readiness；实机模式可显式传入 `start_game=true, allow_launch=true, start_debugger=true, run_probes=true`，让工具执行“启动/连接 -> readiness -> profile-gated 动态 probe”的完整链路。

## 风险与边界

- 战场画面中的颜色、按钮和提示可用于操作，但不能单独作为内存结构结论。
- Data 号、战场编号和画面角色名的互转必须另行验证；在完成编号体系闭环前，操作指令应使用画面可确认的人物名、位置和按钮，而不是只使用内存推断的人名。
- 当前还没有稳定的屏幕坐标自动点击器。自动化前必须先把单位坐标、镜头偏移、菜单状态和按钮位置都转成可校验状态。
- x32dbg 的暂停会让游戏窗口无响应；没有必要时优先使用运行中快照。
- 人工操作必须避免连做多个动作，否则 diff 会混入敌军行动、UI 刷新、回合状态和临时文本。

## 2026-06-11 自动阶段调试口径

测验1续确认：当当前战场可控我军只剩曹操一人时，曹操完成攻击后没有其它可控角色，游戏会自动进入弹窗/阶段切换，并依次进入友方阶段、敌方阶段，最终回到第二回合我方可控阶段。友方阶段和敌方阶段玩家不能控制，但物理攻击、反击、HP 写入仍走同一套伤害应用链。

调试准备：

- 自动阶段前保留 `0x00405AD5` 暂停断点，用于捕获每段物理伤害。
- `0x0043F70C` 可设为 log-only，记录最终 HP 写入：`ECX=unit*`、`EDX=new_hp`、`idx=(ECX-0x004A7B20)/0x30`。
- 记录 `0x004927F0+0x00/+0x01`，不要假设攻击者总是玩家；友军、敌军和反击都会改写当前伤害方向。
- 当前画面回到我方可控阶段后，再继续布设策略、经验、功勋写断点，避免自动阶段高频 HP 写入淹没低频字段。

自动化状态机应至少区分：

| 状态 | 玩家可控 | 调试重点 |
|------|----------|----------|
| 我方可控阶段 | 是 | 操作按钮、移动/攻击/策略入口、单动作采样 |
| 友方自动阶段 | 否 | 物理攻击、反击、友方 AI 行动链 |
| 敌方自动阶段 | 否 | 敌军攻击、我方反击、击杀派生 |
| 回合切换/弹窗 | 否或受限 | 回合开始、行动位恢复、二次行动状态重置 |

### 2026-06-11 阶段性自动化能力评估

经过测验1和测验1续的长时间采样，当前可以实现的是“只读状态识别 + 半自动调试采证 + 有限操作辅助”，还不能称为稳定的“无人战场操控器”。

已具备的能力：

- 可只读识别战场单位数组：`0x004A7B20`、stride `0x30`、当前 HP `unit+0x10`。
- 可通过 `0x004927F0` 读取当前攻击者、目标、伤害缓存、暴击、连击/循环和反击/嵌套行动标志。
- 可用 `0x00405AD5 -> 0x0043F6DB -> 0x0043F70C` 稳定观测物理伤害应用和 HP 写入。
- 可区分玩家主动攻击、友军/敌军自动阶段和反击链的主要调用栈形态。
- 可在曹操主动攻击和部分反击/自动阶段样本中观测个人 EXP、武器 EXP、防具 EXP 的战斗上下文缓存写入。
- 可通过人类操作员执行动作，调试侧自动记录断点、调用栈、内存和知识库证据。

尚不能直接全自动操控的原因：

- 没有闭环的“屏幕格坐标 <-> 战场格坐标 <-> 当前镜头偏移”映射；移动和目标点击仍依赖人工画面判断。
- 移动函数和坐标写入只有候选链，缺少“只移动不攻击”的写断点闭环。
- 当前没有可靠识别 UI 子状态的内存状态机：单位已选中、紫色移动格显示、红色攻击范围、命令菜单、攻击目标选择、动画中、弹窗等待等仍未全部内存化。
- 攻击按钮、结束回合弹窗和策略/道具菜单的按钮坐标虽有人工经验，但缺少跨窗口大小和不同 UI 状态的防误点校验。
- 连击判定、命中/闪避/格挡、反击谓词、策略结算、行动结束和回合开始仍有关键地址未闭环，无法安全地把战斗 AI 和 Hook 注入做成通用规则。
- 五维功勋实际存储结构仍未定位；旧候选已证伪或改判，不能用功勋作为自动化奖励反馈。

当前可交付的自动化形态：

| 等级 | 可行性 | 说明 |
|------|--------|------|
| 只读战场监控 | 可实现 | 自动读取单位、HP、阶段差分、伤害事件和断点证据 |
| 人机协同采证 | 可实现 | 用户操作，调试侧自动下断点、采集、归纳、落库 |
| 有限脚本辅助点击 | 局部可做 | 只适合固定窗口、固定关卡、固定状态；必须带防误点门禁 |
| 内部函数级操控 | 尚未实现 | 需要定位战场 action dispatcher、移动/攻击 ABI 和 UI/脚本状态变量 |
| 通用无人战场 AI | 尚不可交付 | 还缺状态机、路径规划、目标选择、输入可靠性、异常恢复和跨版本定位 |

下一阶段最小闭环：

1. 用单独样本闭环移动坐标写入：移动前后只做移动，不攻击。
2. 建立内存化 UI 状态机：空闲、选中、菜单、目标选择、动画、弹窗、自动阶段。
3. 固定 640x440 客户区下做有限点击器，并把每次点击前后的单位数组和 UI 状态作为门禁。
4. 对 `on_action_end`、`on_turn_start`、`on_hit_check`、`on_double_attack_check`、`on_counter_check` 补齐成功/失败样本。
5. 在能可靠回到同一存档/同一战场状态后，再考虑无人批量采样和自动战斗策略。

### 2026-06-11 测验1调试方法沉淀：跨版本/跨游戏复用流程

本流程来自曹操 `unit[0]` 攻击黄巾兵 `unit[77]` 并触发连击的实机调试闭环。它的可复用价值不在固定地址，而在采证顺序。

1. 先定位运行时实体数组，而不是先找公式。用内存快照/差分确认单位数组基址、单位步长、HP 字段和至少两个可画面确认的实体。
2. 对真实变化字段下硬件写断点。本次是目标 HP 地址 `0x004A89A0`，命中后才确认最终写入指令 `0x0043F70C`。
3. 从写入点向上追调用栈。本次闭环为 `0x0043F70C <- 0x0043F6DB <- 0x00405ADC <- 0x00405966 <- 0x00418330`。
4. 再把调用栈与用户动作绑定。本次动作入口闭环到 `0x0043DF0A -> 0x0043DADA`，且 `ECX=0x004A7B20` 对应曹操。
5. 最后区分“结果应用”和“结果生成”。`0x00405966` 能解释为什么连击会两次写 HP，但不能替代连击判定或伤害公式。

跨版本/跨游戏迁移时不要照搬绝对地址，应迁移以下锚点：

- 实体数组：基址、stride、实体有效标志、画面可验证身份。
- 字段锚点：当前 HP、坐标、阵营、行动状态等最容易由画面行为确认的字段。
- 写入闭环：硬件写断点、写入指令、setter、上层结果循环。
- 动作闭环：玩家操作入口、命令分支、核心调度函数。
- 结果缓冲：伤害段数、目标编号、旧值/新值、动画/事件派发。

禁止外推：

- 未做移动样本时，不得把坐标候选字段或攻击链函数写成移动函数。
- 只命中 HP 写入时，不得把 setter 或结果循环写成伤害公式。
- 只观察到两次 HP 写入时，不得把“连击判定函数”写成已确认；必须追到第二段伤害记录的生成点。
- UI 事件、动画、事件派发和状态恢复函数不能因为出现在调用栈附近就标记为 HP 写入点。
