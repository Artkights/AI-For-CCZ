# 特效工作台复盘与验收

## 结论速览

- 特效工作台的完成标准固定为：`Active 启动 -> 扫描 -> 选择 -> 理解证据 -> 编辑 -> 预览 -> 写入测试副本 -> 复读 -> 恢复 -> SHA 一致`。
- 服务层可以扫描或生成补丁，不等于桌面工作流已经可用；真实窗口响应、布局、权限一致性和写后复读是同等重要的验收项。
- 当前工作台已收敛为“特效清单”和“复合制作”两个页面；原生配置直接附着在扫描结果上，不再要求用户手工输入编号重新读取。
- 行为逻辑只有同档案、同代码身份、同契约的 V3 动态证据通过后才能正式写入；高级模式不能绕过。

## 适用版本

- 默认适用：曹操传加强版 6.5（未加密）规范基底 `F9A357585F4D77C273A8BADD3E4EF4062983CC81D8040C0CB74768FAB7C04F5F`。
- `84E3A1DC085AE6F9900D1E8C388A9CD6766379832DDF51BC7BDF780C6615B4A3` 是已登记数据型变体，可以复用未受影响的代码契约，但不能借用另一代码身份的行为证据。
- 6.6、未知 EXE 和未入库的 6.5 变体继续只读；未知文件只能在自动测试副本中验证。

## 已确认事实

- 旧顶部“权限中心”重复显示标题栏已有的 BuildIdentity，并长期占用三个低频命令，现已删除；验证命令进入“验证”菜单。
- 旧“当前特技”和“原生配置”描述同一对象却使用两套页面。现统一为“特效清单”，默认显示全部，并提供原生、注入、复合、研究中筛选。
- 原生配置旧布局将剩余高度按 `35%/65%` 分给说明和影响范围，绑定表被压缩。现改为稳定行高、可滚动绑定表和按所选渠道自动读取。
- 旧复合扫描在 UI 线程同步执行，并对每个成员重复构建机制档案和兼容性。现改为共享 `EffectAnalysisSnapshot`、一次机制档案、一次批量兼容和后台可取消加载。
- Windows 在 `2026-07-15 21:57:16` 和 `22:03:52` 记录 `AppHangB1`；这两次是窗口无响应后被关闭，不是普通业务异常。
- GameDebug 客户端断开时曾在管道响应写入处抛出 `IOException: Pipe is broken`。现将断开作为当前连接结束，宿主继续接受下一次请求。
- 当前 UI 烟测会实际打开隔离的 WinForms 特效页、点击复合扫描，并以高优先级 UI 回调测量心跳；P99 心跳和每个绑定阶段均不得超过 250ms，单个投递合并产生的最大间隔只保留作诊断，同时断言成员表非空。

## 实现/使用方法

1. 正式版本只使用 `MCP配置/start-ccz-studio.ps1` 启动；脚本会验证 `CCZModStudio_Releases/Active` 的发布清单和组件摘要。
2. 开发验证统一使用 `MCP配置/start-ccz-studio-dev.ps1`；默认构建并启动 Debug，也可选择 Release、`-NoBuild` 或 `-DryRun`，脚本会固定传入 `--developer-build`。
3. 不要把直接双击 `bin/Debug` 或 `bin/Release` 当作开发验证：未带开发参数时，源码构建会按正式策略自动转到 Active。
4. 在“特效清单”扫描后按来源筛选；选择结果即可查看人物/兵种和宝物原生配置、参数、证据、阻断码和维护动作。
5. 字段是否可输入、可预览和可写入全部以 `EffectWriteDecision` 为准。
6. “高级诊断”复用当前完整快照，并展示确认台账和物理位置，不触发第二次反汇编。
7. “复合制作”后台完成分析；不可加入成员仍显示在列表中，并给出包装入口、家族契约或来源不唯一等原因。
8. 每次写入先预览，再明确确认；写入后自动失效缓存、复读当前配置并保留事务 manifest。

### 唯一主验收链

```text
Active 启动
  -> 扫描全部特效
  -> 选择一个扫描结果
  -> 查看物理/结构/语义/动态证据
  -> 只编辑 CanEdit 字段
  -> 生成锁定预览
  -> 写入自动测试副本
  -> 重新扫描和复读
  -> 删除或恢复
  -> 校验最终 SHA
```

## 风险与边界

- 不再用“服务类返回成功”“构建通过”或“能生成汇编”替代真实窗口验收。
- 不再为同一对象增加平行页面和独立权限文案；桌面、Core 和 MCP 必须读取同一决定。
- 不把自然语言名称、历史相似样本或单次动态命中写成完整业务语义。
- 不因复合现有特效生命周期通过，就宣布恢复、状态、行动结束、回合开始或相邻目标行为已经开放。
- UI 心跳测试需要隔离主窗体启动任务，否则整包自检和项目自动发现会污染特效扫描测量。

## 反思与强制改进规则

- 过去把大量时间投入静态分析、安全契约和服务层烟测，却没有把用户点击路径作为完成标准，导致“后端越来越强，界面仍不可用”。
- 三个重叠页面让同一特效出现不同名称、不同权限和不同操作入口；以后一个业务对象只能有一个主视图模型和一个权限来源。
- 静态识别被展示得过于接近真实语义；以后界面必须明确区分物理确认、结构确认、语义推断和动态确认。
- 发布入口不唯一使源码构建和 Active 看起来相同；以后正式版本必须从 Active 启动，开发构建必须显式标识。
- 新功能在完成主验收链前不得继续横向扩展。优先修通一个真实闭环，再增加下一个行为家族。
