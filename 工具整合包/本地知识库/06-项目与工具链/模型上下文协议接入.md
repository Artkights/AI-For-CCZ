# 16 MCP 接入

## 结论速览

- MCP Server 当前只保留制作所需的无头读写工具：项目识别、数据表、R/S 剧本、Hexzmap、地图底图、图片资源、特效包和本地知识库。
- 2026-06-24 起，跨文件 MOD 制作优先走自动化 `ModPackage` 流水线：自然语言需求分析、自动槽位规划、整包预览、direct 写入当前项目、自动验证、失败修复和作者报告。
- 非制作主线的管理、检查、留痕、发布和自动恢复类入口已移出当前 MCP 工具集。
- 写入类操作仍必须以 6.5 未加密基底、本地样本、写前备份、写后复读和结构化报告为最终依据。

## 适用版本

- 默认适用：曹操传加强版 6.5（未加密）当前项目基底。
- 6.6/6.6x 或旧版本只作为版本差异、兼容提示或外部佐证，不自动进入 6.5 写入规则。

## 已确认事实

- `CCZModStudio.McpServer` 是本地 stdio MCP Server，引用 `CCZModStudio.Runtime`，不启动 WinForms 界面。
- MCP Server 启动后只通过 stdout 发送 MCP JSON-RPC 消息；普通诊断输出不得写 stdout。
- 高风险写入继续复用桌面端服务层，不绕过路径边界、版本护栏、备份、复读和报告。
- 2026-06-09 已接入 x32dbg 调试通道：本机 x64dbg snapshot `2026.05.27` 安装在 `CCZModStudio_Exports/DebugTools/x64dbg`，`x64dbg_mcp.dp32` v2.3.0 已放入 `release/x32/plugins`，项目级 `.codex/config.toml` 已包含 `cczmodstudio` 与 `x64dbg` 两个 MCP Server。

## 实现/使用方法

- 推荐通过 `工具整合包\MCP配置\start-ccz-mcp.ps1` 启动，必要时设置 `CCZMODSTUDIO_GAME_ROOT` 固定项目根目录。
- 客户端配置由 `工具整合包\MCP配置\generate-mcp-config.ps1 -Build` 生成；若同时启用 x32dbg MCP，增加 `-IncludeX64dbg`。
- 打开 6.5 未加密基底进行动态调试可用 `工具整合包\MCP配置\start-x32dbg-6.5.ps1`；打开后 x32dbg 日志应出现 MCP 插件监听 `127.0.0.1:27042`。
- 静态地址复读和断点批次生成可用 `工具整合包\MCP配置\capture-6.5-debug-baseline.ps1 -GameRoot ".\基底\加强版6.5未加密版"`，输出到 `CCZModStudio_Reports\DebugEvidence`。
- 软件断点批量布设可用 `工具整合包\MCP配置\apply-x32dbg-breakpoints.ps1`；继续运行直到计划断点命中可用 `工具整合包\MCP配置\run-x32dbg-until-planned-hit.ps1`。高频启动断点可先加 `-DisableAddress 00484002` 或在暂停时用 `-DisableCurrentPlannedHit`，仅禁用调试断点，不修改 EXE。
- 修改游戏文件前必须确认目标版本、目标路径、写前备份、写后复读和结构化报告。

## 风险与边界

- MCP 只保留制作读写工具；非制作主线的管理、检查、留痕、发布和自动恢复能力不作为当前入口。
- 恢复只依赖写入服务生成的备份文件手动处理；MCP 不提供自动恢复入口。
- `replace_resource` 禁止直接覆盖 `Ekd5.exe`、`Data.e5`、`Imsg.e5`、`Star.e5`、`Hexzmap.e5` 等核心文件；这些文件必须走专用写入工具。
- 所有项目内目标路径都会做根目录边界校验，拒绝 `..` 路径逃逸。
- x32dbg MCP 只用于调试采证：断点、寄存器、内存、反汇编、trace、AOB/string 搜索和证据导出。不得把 x32dbg 的内存写入、patch 工具当作正式补丁写入入口；正式 EXE 改动仍走 CCZModStudio 的补丁预览、备份、复读和报告。
- 若启用 `X64DBG_MCP_TOKEN`，只允许存放在本机 Codex 配置或环境变量，不得写入仓库文档或脚本。

安全边界：GameDebug MCP 不写游戏文件、不 patch EXE、不写目标进程内存；x32dbg bridge 只允许本机回环地址，默认 `127.0.0.1:27042`，可用 `X64DBG_MCP_TOKEN` 作为本机 token。所有证据默认写入 `CCZModStudio_Reports\DebugEvidence\game-auto-*`，动态命中必须结合具体游戏阶段、CIP、寄存器、调用栈、反汇编和战场状态差异复核。

证据文件口径：

- `events.jsonl` 是低层诊断流，记录工具内部步骤和 x32dbg 调用结果。
- `action-chain.jsonl` 是高层动作审计流，由 `debug_full_auto_run`、`debug_runtime_invoke_run`、`debug_menu_route_run`、`debug_address_verify_run` 等入口写入；每行包含工具、动作、状态和可选前后内部状态，作为“最终不依赖鼠标/截屏”的验收主线。
- `probe-hits.jsonl` 是地址验证索引。plan-only 运行会写计划/summary 行以固定证据路径；live 断点命中时会追加 `breakpoint-hit` 行，指向 `x32dbg/internal-probe-hit-*.json` 的寄存器、栈、反汇编、内存和战场状态证据。
- `knowledge-promotions.json` 只描述可推广/待验证状态；正式知识库追加必须由 `debug_knowledge_promote(allow_write=true)` 且全部动态证据门槛通过后执行。

#### 写入边界

- 自动化 MOD 制作的通用执行规则见 `自动化MOD制作.md`。独立单关或独立精简基底必须先完成设计定稿、槽位复读、资源门控和预览报告，然后由专用写入器 direct 写入当前项目。
- `ModPackage` 是多文件 MOD 的正式 AI 输出格式。自动化优先入口为 `auto_make_mod`：自然语言需求会被解析成 `ModDesign`，再自动编译为 `ModPackage`，preview，通过后直接写入当前项目，失败时自动修复 GBK 超长、ID 冲突、资源缺失和 effect 未绑定等常见问题并重试。
- 手动编排时先用 `analyze_mod_request` / `compile_mod_package`，再提交 `preview_mod_package`，最后执行 `apply_mod_package(automation_mode=direct)` 或独立单关的 `apply_mod_package(automation_mode=force_open)`。`write_mode` 只作为旧客户端兼容输入，测试副本语义会被忽略。
- `preview_scenario_patch` / `apply_scenario_patch` 继续提供保守参数补丁。自动化剧本制作使用 `compile_scenario_patch` / `apply_scenario_patch_aggressive`，结构化 `append_command` / `insert_command` 会通过 `LegacyScenarioWriter` 重建并复读。AI 剧本文本宏可走 `apply_scenario_text_import` 直接编译写回；R 场景视觉草稿可走 `publish_rscene_draft_to_scenario`。
- `write_table_rows` 按 `row_id` 写入指定列，拒绝 `ID` 和派生列。
- `write_scenario_texts` 只按文本 `index` 原地写回，必须满足原 GBK 容量。
- `write_hexzmap_block` 只按 `Mxxx` 和坐标写入 Hexzmap 地形格，继续受版本护栏约束。
- `replace_map_image` 只替换 `Map\Mxxx.jpg` 地图底图。
- `replace_e5_image_entry` / `replace_e5_image_batch` 只允许可读取 `110` 索引表的 E5 图片载荷文件，拒绝核心表文件。
- `replace_dll_icon` / `clear_dll_icon` 只开放 `Itemicon.dll`、`Mgcicon.dll`、`Cmdicon.dll` 的 RT_BITMAP 位图资源。
- 人工智能 图片工具默认写 `CCZModStudio_Exports\AiImageAssets` 和预览产物；需要直接落入 E5/DLL 资源时走 `draw_and_replace_ccz_image_asset`，仍保留备份、报告和复读校验。
