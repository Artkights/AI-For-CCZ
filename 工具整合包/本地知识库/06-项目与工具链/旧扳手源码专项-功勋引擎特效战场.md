# 旧扳手源码专项：功勋、引擎、特效与战场

## 结论速览

- `老版游戏制作工具\扳手` 源码是 Python + PyQt5 运行时内存调试器，核心业务集中在 `mywindow.py`，远程调用集中在 `hook.py`。
- 旧扳手源码中的功勋表为独立运行时表：6.4 走 `0x508000`，其它旧版走 `0x508400`；每人 `10` 字节，五个 2 字节字段，顺序为武力、统率、智力、敏捷、运气。
- 旧扳手识别的是 6.1、6.2、6.3、6.3mp+、6.4 和 6.4 状态修正版/圣三分支；当前项目默认 6.5 未加密，旧源码地址只能作为定位线索，不能直接作为 6.5 写入规则。
- 旧扳手把“天赋”作为个人/兵种/宝物特效的运行时配置页：`addr_tianfu` 维护三个人物、一个兵种和一个效果值；`addr_zhuanshu` 维护两组专属和两组套装。
- 战场单位状态和属性升降直接写 `addr_war + war_code * len_war` 的单位记录；属性升降字段是 `+0x18..+0x1D`，旧扳手用 `1=下降`、`3=正常`、`6=上升`。
- 控制页会直接 patch 旧引擎代码地址，例如敌友军可控、不分敌我、禁 AI、待命后可移动、移动力 255、穿越移动、自动回归、托管我军主动出击和我军自动复活。

## 适用版本

- 源码证据适用：旧扳手支持的 6.1 到 6.4 系列运行时。
- 当前项目用途：为曹操传加强版 6.5 未加密基底提供源码级交叉参照、候选地址和字段命名。
- 禁止用途：不得把本页旧版运行时地址直接写入 6.5 工具、补丁包或自动化脚本；迁移必须重新做 6.5 本地样本、x32dbg 动态命中、复读或烟测。

## 已确认事实

| 文件 | 作用 |
|------|------|
| `mywindow.py` | 主窗口、进程附加、版本识别、人物/战场/仓库/天赋/变量/必杀读写、控制页代码 patch。 |

### 引擎识别与版本几何

旧扳手拖动准星附加目标进程后，先读 `0x40100E`，值为 `0x774` 时认为是曹操传；也接受窗口类名为“豪华曹操传”的进程。自动版本识别读取 exe 文件版本低字：

| 版本识别 | `version` | 主要几何 |
|----------|-----------|----------|
| 文件版本低字 `4` | `0`，6.4 | `addr_war=0x4A7B20`、`len_war=0x30`、我/友/敌容量 `20/40/190`、物品上限 `255`、复活入口 `0x4092C7`、必杀表 `0x511800` |
| 文件版本低字 `3` | 先为 `2`，6.3；若 `0x41A8A0` 首字节为 `0x55`，改为 `1` 即 6.3mp+ | `addr_war=0x4B2C50`、`len_war=0x24`、容量 `16/19/80`、物品上限 `154`、复活入口 `0x4092E0`、必杀表 `0x508800` |
| 文件版本低字 `2` | `3`，6.2 | 同 `version >= 1` 主体几何 |
| 其它 | `4`，6.1 | 同 `version >= 1` 主体几何，但复活入口改为 `0x40930F` |
| `0x401007 == 0xF4E9` | `-1`，6.4 状态修正版/圣三分支 | 禁用部分页面，仅保留控制页口径 |

其它版本相关表：

| 表 | 地址选择 |
|----|----------|
| 专属/套装表 | `version >= 3` 用 `0x50E800`，否则用 `0x50E400` |
| 天赋表 | `version == 0` 用 `0x508998`，否则用 `0x5089B0` |
| 6.4 状态修正版判定 | `version == 0` 且 `0x4240CA == 0xFC` 时启用 `condition_change=True`，异常状态字段按四个 2-bit 小字段解释 |

### Data、角色、兵种、装备读取

旧扳手的人物数据基址不是硬编码表起点，而是先读指针 `0x4CEA00`，再按 `base + id * 0x48` 访问角色记录。

| 数据 | 读取方式 |
|------|----------|
| 角色名 | `*(0x4CEA00) + id * 0x48 + 0x08`，最多 8 字节 GBK |
| R 形象 | `0x50F800 + id * 2`，2 字节 |
| S 形象 | `0x501000 + id * 2`，2 字节 |
| 头像 | `0x50F000 + id * 2`，2 字节 |
| 兵种名 | `0x5000D0 + job_id * 0x09`，每项 9 字节 |
| 宝物名 | `0x4A1140 + item_id * 25`，前 12 字节名称，`+0x14` 是类型过滤字段 |
| 消耗品名 | `version >= 1` 用 `0x4A19BF + id * 25` 共 43 项；6.4 用 `0x4A1FE6 + id * 25` 共 105 项 |
| 装备仓库 | `0x4B0783 + slot * 3`，三字节分别是物品、等级、经验 |
| 金钱 | `0x4B077C`，4 字节 |
| 忠奸度 | `0x4B0782`，1 字节 |
| 功勋池 | `0x505F40`，4 字节 |

人物记录中常用偏移：

| 偏移 | 含义 |
|------|------|
| `+0x11/+0x13/+0x15/+0x17/+0x19` | 攻击、防御、精神、爆发、士气，均 2 字节 |
| `+0x1B` 或 `+0x1C` | HP 上限；`version <= 1` 用 `+0x1B`，否则用 `+0x1C`，4 字节 |
| `+0x1F` 或 `+0x20` | MP 上限；`version <= 1` 用 `+0x1F` 2 字节，否则用 `+0x20` 1 字节 |
| `+0x21..+0x25` | 武力、统率、智力、敏捷、运气档位/成长 |
| `+0x27/+0x28/+0x29` | 出场、幸存、撤退 |
| `+0x2A` | 我军标记；旧扳手勾选我军写 `0`，非我军写 `255` |
| `+0x2B` | 兵种 |
| `+0x2C/+0x2D` | 等级、经验 |
| `+0x2E/+0x2F/+0x30` | 武器、武器等级、武器经验 |
| `+0x31/+0x32/+0x33` | 防具、防具等级、防具经验 |
| `+0x34` | 辅助装备 |
| `+0x35` | 击杀数，2 字节 |
| `+0x06/+0x44/+0x46` | 旧扳手作为原始 2 字节字段显示，源码没有进一步命名 |

### 角色功勋地址

旧扳手人物页读取/保存五维功勋时，先按版本选择表基址：

| 条件 | 功勋表基址 |
|------|------------|
| `version == 0`，即 6.4 | `0x508000` |
| 其它旧版 | `0x508400` |

功勋表只对 `id <= 101` 启用。每个角色占 `10` 字节：

| 字段 | 地址 |
|------|------|
| 武力功勋 | `gongxun + id * 10 + 0`，2 字节 |
| 统率功勋 | `gongxun + id * 10 + 2`，2 字节 |
| 智力功勋 | `gongxun + id * 10 + 4`，2 字节 |
| 敏捷功勋 | `gongxun + id * 10 + 6`，2 字节 |
| 运气功勋 | `gongxun + id * 10 + 8`，2 字节 |

当前 6.5 已有实机调试结论说明五维功勋实际存储仍未闭环，且此前候选 `0x00492F9C/0x00492C20/0x004B1B1A/0x00496E79` 已被证伪或改判。因此本页旧扳手功勋表只能作为“旧版工具源码地址”，不能推翻当前 6.5 的待验证状态。

### 天赋、专属、套装读取

旧扳手“天赋”页实际覆盖个人/兵种/宝物特效配置：

| 表 | 结构 |
|----|------|
| 天赋表 `addr_tianfu + row * 8` | `+0/+2/+4` 为三个人物 id，均 2 字节；`+6` 为兵种 id，1 字节，`255` 表示空；`+7` 为效果值，1 字节 |
| 专属/套装表 `addr_zhuanshu + row * 16` | 专属 1：人物 `+0` 2 字节、物品 `+2` 1 字节、效果 `+3` 1 字节 |
| 同上 | 专属 2：人物 `+4` 2 字节、物品 `+6` 1 字节、效果 `+7` 1 字节 |
| 同上 | 套装 1：物品 `+8/+9/+10`，效果 `+11` |
| 同上 | 套装 2：物品 `+12/+13/+14`，效果 `+15` |

保存时的特殊规则：若专属或套装的效果值为 `0`，旧扳手会把对应人物/物品字段清零；界面中的“空”在部分下拉框映射为 `255`，不是所有空字段都写 `0`。

天赋/必杀名称表按版本读取：

| 版本 | 天赋名表 | 必杀名表 |
|------|----------|----------|
| 6.4 | `0x4FF560`，180 项，每项 16 字节 | `0x4A7510`，80 项 |
| 6.3/6.3mp+ | `0x4FF7A0`，144 项 | `0x4FF551`，36 项 |
| 6.2 | 源码尝试在 `0x4F0000` 与 `0x4FF960` 间判断，存在 `i/j` 未定义缺陷 | `0x4FF710`，36 项 |
| 6.1 | `0x4FFA80`，88 项 | `0x4FF710`，36 项 |

### 战场单位状态和属性升降

旧扳手战场页以 `war_base = addr_war + war_code * len_war` 读写单位。`war_code` 由当前选择的阵营列表和版本容量换算：我军从 `0` 开始，友军从 `cnt_wo` 开始，敌军从 `cnt_wo + cnt_you` 开始。

| 偏移 | 含义 |
|------|------|
| `+0x00` | Data 编号，2 字节；`0xFFFF` 表示空槽 |
| `+0x04` | 出场/显示编号，旧扳手用于战场列表显示 |
| `+0x05` | 阵营/队伍标记 |
| `+0x06/+0x07` | 当前 X/Y |
| `+0x08` | AI 目标单位索引；`255` 表示使用坐标目标 |
| `+0x09/+0x0A` | 目标坐标 X1/Y1 |
| `+0x0C` | 显示/死亡状态；HP 为 0 且该字段为 3 时启用复活按钮 |
| `+0x0D` | 行动状态；保存时勾选“行动”写 `old | 6`，取消写 `old & 248` |
| `+0x0E` | AI/方针；`3=被动出击`、`1=主动出击`、`2=坚守原地`、`0=攻击武将/到指定点`、`4=跟随武将/逃到指定点`、`7=友军可控` |
| `+0x0F` | 方向 |
| `+0x10` | 当前 HP，4 字节 |
| `+0x14` | 当前 MP，4 字节 |
| `+0x18` | 攻击临时升降 |
| `+0x19` | 防御临时升降 |
| `+0x1A` | 精神临时升降 |
| `+0x1B` | 爆发临时升降 |
| `+0x1C` | 士气临时升降 |
| `+0x1D` | 移动临时升降 |
| `+0x1E` | 异常状态；普通模式按 bit，状态修正版按四个 2-bit 字段 |
| `+0x20/+0x21` | 第二目标坐标 X2/Y2 |
| `+0x22/+0x23` | 旧扳手原样显示的扩展字段 |

属性升降读写规则：

| UI 状态 | 写入值 |
|---------|--------|
| 下降 | `1` |
| 正常 | `3` |
| 上升 | `6` |

单体修改直接写 `war_base + 0x18..0x1D`。群体 buff/debuff 先按阵营选择范围：我军 `[0, cnt_wo)`，友军 `[cnt_wo, cnt_wo + cnt_you)`，敌军 `[cnt_wo + cnt_you, cnt_wo + cnt_you + cnt_di)`，再批量写同一字段。

普通异常状态模式写 `war_base + 0x1E`：

| 状态 | bit 值 |
|------|--------|
| 麻痹 | `0x02` |
| 禁咒 | `0x04` |
| 混乱 | `0x08` |
| 中毒 | `0x10` |

6.4 状态修正版/圣三分支启用后，`+0x1E` 改按四个 2-bit 小字段组合：`v0 + v1*4 + v2*16 + v3*64`。

战场全局字段：

| 字段 | 地址 |
|------|------|
| 回合上限 | `0x4B3D0B` |
| 当前回合 | `0x4B3D0A`，UI 显示时加 1 |
| 下一关 | `0x4B0776`，战斗中写 `stage * 2 - 1` |
| 天气主字段 | `0x4B3D0C`，晴/阴/小雨/大雨/雪分别写 `0/3/4/5/0x16` |
| 天气辅助字段 | `0x4B3D19`，旧扳手改天气后清零 |
| SP | `0x501C00 + data_code`，不在单位记录内 |

### 控制页 patch 地址

旧扳手控制页 `onButtonClick(n)` 直接修改运行时代码或战场字段：

| 功能 | 启用写入 | 关闭还原 |
|------|----------|----------|
| 可控制敌军和友军 | `0x43D574=EB`、`0x43D5A5=EB`、`0x43E359=EB`、`0x44E748..49=90 90`、`0x44E754..55=90 90`、`0x455379=EB` | `0x43D574=74`、`0x43D5A5=74`、`0x43E359=74`、`0x44E748..49=75 C8`、`0x44E754..55=75 BC`、`0x455379=74` |
| 不分敌我 | `0x4242B0=90`，6.4 还写 `0x4242AF=90`；`0x437BC0..C1=90 90`、`0x454405..06=90 90`、`0x455646=EB` | `0x4242B0=32`；6.4 写 `0x4242AF=74,0x4242B0=DD`；`0x437BC0..C1=74 34`、`0x454405..06=75 27`、`0x455646=75` |
| 禁止 AI 行动 | `0x44E9D3=EB` | `0x44E9D3=74` |
| 待命后可移动 | `0x43D5DC=EB` | `0x43D5DC=74` |
| 移动力 = 255 | `0x43F74F..52 = 33 C0 48 C3` | `0x43F74F..52 = 55 8B EC 83` |
| 穿越移动 | `0x4387B7=EB`、`0x437D59..5B = 33 C0 C3` | `0x4387B7=74`、`0x437D59..5B = 55 8B EC` |
| 自动回归 | 修改 `0x44E571..74` 或 `0x44E576..79`、`0x44EA74..77`，并在 `0x485E00` 起写入一段按 `len_war/addr_war` 计算的 stub | 按版本还原 `0x44E571..74` 或 `0x44E576..79`、`0x44EA74..77` |
| 托管我军主动出击 | 遍历我军和友军非空槽，把 `unit+0x0E` 写 `1` | 遍历同范围，把 `unit+0x0E` 写 `7` |
| 我军自动复活 | 不 patch 代码，只设置 `auto_life=True`，后台线程每秒扫描我军 HP=0、非空、`+0x0C=3` 的单位并调用复活远程函数 | `auto_life=False` |

自动回归功能的 stub 写入点 `0x485E00` 是旧扳手运行时热补丁代码洞，源码把 `len_war` 和 `addr_war` 嵌入新逻辑。迁移到 6.5 时必须重新找代码洞、原字节、跳转距离和回滚方案。

### 远程调用地址

`hook.py` 的调用方式是把 shellcode 和参数区写入目标进程，再用 `CreateRemoteThread` 执行。旧版内部地址如下：

| 函数 | 用途 | 内部地址 |
|------|------|----------|
| `life()` | 战场创建/复活单位 | 由主程序按版本传入：6.4 为 `0x4092C7`，6.3mp+/6.3/6.2 为 `0x4092E0`，6.1 为 `0x40930F` |
| `recal()` | 重新计算人物能力 | `0x4075DF` |
| `changeDir()` | 调整朝向并刷新 | `0x4B5DF0`、`0x457428` |
| `changePos()` | 调整坐标并刷新 | `0x4B5DF0`、`0x4594DD`；源码中未见 UI 入口直接调用 |
| `changeWeather()` | 改天气并刷新 | 设置 `ECX=0x4B3D08`，调用 `0x41D9D1` |

## 实现/使用方法

1. 查旧扳手源码时，先定位版本：`mywindow.py` 的 `version` 决定战场数组、记录长度、容量、天赋表、专属表、必杀表和复活函数。
2. 查功勋时，只能把 `0x508000/0x508400` 当旧 6.x 运行时表；当前 6.5 必须继续按 `功勋模式与五维功勋.md` 的写断点流程重新定位。
3. 查个人/兵种/宝物特效时，从“天赋页”读起：天赋表负责人物/兵种绑定，专属/套装表负责宝物组合和效果值。不要把旧扳手的“天赋”字面理解成只有个人天赋。
4. 查战场状态和属性升降时，以 `addr_war + war_code * len_war` 为单位记录，优先核对 `+0x10` HP、`+0x14` MP、`+0x18..+0x1D` 属性升降、`+0x1E` 状态、`+0x0D` 行动位、`+0x0E` AI/可控。
5. 迁移控制页功能时，先把旧 patch 地址作为断点选址线索，再在 6.5 上做原字节确认、动态命中、调用栈、行为样本和回滚方案；不得静态照搬。

### 2026-06-13 工具整合包落地口径

- 旧扳手的“按版本选择地址/表几何”思想已迁移为离线工具的 `CczEngineProfileService`：不读取旧运行时地址，而是按当前项目 `Ekd5.exe` 尺寸、版本资源和路径提示识别 6.3/6.4/6.5/6.6，再输出 `TableHints` 给 Data、人物、物品、兵种、商店和特效读取使用。
- 当前工具整合包对人物、装备、详细兵种、兵种系、宝物特效名称、兵种特效说明/分配、个人/套装专属和商店数据统一走 `HexTableNameResolver.ResolveForProject`；缺少对应版本离线表时只允许语义兜底读取，不把旧扳手运行时地址当写入依据。
- 旧扳手“天赋”页的专属/套装清空规则已落实到 `EffectPackageService`：非删除模式只写 EffectPackage 明确指定的槽位；删除模式才清空 `6.X-7-3 人物专属、套装专属` 固定 14 列。指定槽位的特效值为 `0` 时，按旧扳手习惯清空该槽人物/物品/效果值字段。
- 战场状态候选下拉读取人物、详细兵种和物品名时已改为项目引擎感知解析；但旧扳手 `addr_war` 字段和控制页 patch 地址仍只作为运行时调试线索，不进入离线文件写入规则。

## 风险与边界

- 本页是旧扳手源码专项归纳，不是当前 6.5 地址表。
- 当前 6.5 已确认战场单位数组 `0x004A7B20`、stride `0x30`、当前 HP `unit+0x10` 与旧 6.4 几何接近，但这不证明其它字段、函数、功勋表、天赋表或控制页 patch 地址也一致。
