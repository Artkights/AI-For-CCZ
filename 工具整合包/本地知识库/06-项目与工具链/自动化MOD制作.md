# 自动化 MOD 制作

## 结论速览

- 自动化 MOD 制作必须走“设计定稿 -> 槽位复读 -> 预览包 -> direct 写入当前项目 -> 写后复读 -> 实机/烟测 -> 归档报告”的闭环，不能绕过专用写入器裸改文件。
- 独立单关或新工程适合采用“复制独立基底、归档旧 R/S、清空无关定长表行、低位重占用”的路线；定长表和核心文件只清空/覆盖行，不物理删行、不缩短文件。
- 《三英战龙帝》案例沉淀的低位方案为：`R_00/S_00` 正式首关，人物 `0-8`，兵种系 `6.5-3:0-8`，详细职业/详细兵种按每系三行使用 `6.5-4:0/3/6/9/12/15/18/21/24`，武器 `0-8`，防具 `70-78`，辅助 `109-117`，奖励 `118-122`。该方案只适用于独立落实基底，不得直接套到原 6.5 基底。曾试行的 `R_00 -> R_01 -> S_01` 已因剧情跳段、人物显示和末尾路由问题降级为失败尝试。
- R/S 设计必须先查 `08-剧本与战场/剧本设计语义命令映射表.md`，把“单挑、奖励、坐标触发、阶段推进、胜败判定”落到真实命令号、参数槽、变量和写回风险。
- R/S 模板选择必须先分类：独立首关使用三英 `R_00/S_00` 实测路线；既有模组后续关可参考刘备 6.4 `R_01/S_01` 成熟样本；普通文本替换只走文本参数或文本导入宏。不要把某个基底的个人风格当通用规则。
- 地图实现必须区分 `Map/Mxxx.JPG` 底图和 `Hexzmap.e5` 地形格；替换底图不等于可通行地形已完成。
- 头像、R 形象、S 形象和图标资源实行资源门控：没有通过预览和复读前，表内形象字段先写 `0`，不抢占资源号；若复用现有资源，必须先按真实映射生成候选预览，再写 `Ekd5.exe` 指定表。

## 适用版本

- 默认适用：曹操传加强版 6.5（未加密）当前项目和由它复制出的独立落实基底。
- 6.6/6.6x、旧 5.x/6.x 资料只能作为外部佐证；进入自动写回前必须重新做 6.5 样本、工具源码、复读或实机验证。

## 已确认事实

- 当前 MCP / CCZModStudio 自动化主线具备 `auto_make_mod`、`preview_mod_package`、`apply_mod_package(automation_mode=direct)`、`apply_scenario_text_import`、`publish_rscene_draft_to_scenario`、`publish_map_workbench_bundle`、`draw_and_replace_ccz_image_asset`、`validate_mod_package`、`export_mod_report` 等整包流程入口。
- `ModPackage` 适合写人物、职业、物品、资源、特效和 R/S 结构命令；复杂 R/S 结构以 `apply_scenario_patch_aggressive` / `LegacyScenarioWriter` 写入、备份、复读和报告为准。
- 独立基底清理时，`Data.e5`、`Star.e5`、`Imsg.e5`、`Ekd5.exe` 等表结构保持原长度；无关行写成“空 / 预留 / 禁用”，不能物理删除行。
- R/S active 目录可以只保留当前流程需要的 `R_00.eex / S_00.eex`，旧样本和失败尝试进入归档目录；若保留 `R_01/S_01/M001`，应明确标记为参考副本，不作为实机入口。
- `R_00.eex` 在 6.5 原版中是开局启动容器。独立精简基底实测后，不保留原版启动、历史情报、菜单、随机和配置分支；但也不再把正式剧情迁到 `R_01`。当前可靠路线是将 `R_00` 做成纯净首关 R：剧情、人物出场、停顿、入队、装备、初始物品和出战设置在正文 Scene 内，另起玩家切换 Scene 承接出战测试。
- 需要战前出战入口时，优先使用本地样本式双 Scene 链，而不是 `0x72-8 + 0x11`，也不是 `0x07 + 0x11` 硬跳：剧情 Scene 侧写 15 个 `0x4B 我军出场位`，末尾写 `0x08 true`、`0x06 1,3,0,1,2,-1...`；玩家切换 Scene 写 `0x07 出战测试 -> 0x00 -> 0x08 false -> 出发对白 -> 0x06 0,0,-1... -> 0x1D -> 0x24 255 -> 0x0D -> 0x00`。2026-06-20 《三英战龙帝》实机确认该结构可进入战前准备阶段；后续若进入 S 后闪退，应优先排查 S 初始化、部署、地图和数据资源一致性，不得回滚已验证的 R 路由。`0x72-8/4051` 只作为源码线索和历史尝试，不作为自动化模板。
- S 首关生成必须使用样本式完整初始化链。只写 `44/4A/4B/46/47/48/5A` 不够稳；进入 S 后闪退时，优先补查 `45 战场全局变量`、`27 背景显示`、`1C 绘图`、`24 音乐`、`19/1A 胜败条件`、`5A` 后的 `0B` 进度变量和 `08 菜单处理`。三英案例实机确认：缺 `27 背景显示 / 1C 绘图 / 24 音乐` 会导致进入 S 后闪退或显示异常；修复后 `S_00` 保留 `4A -> 44 -> 45 -> 46/47 -> 4B -> 27 -> 1C -> 24 -> 5A -> 08` 链。
- S 战中事件段不能写成“正文根里先 `00` 结束，再放触发条件”。样本式结构应为 `02 内部信息 -> 触发测试命令 -> 05 变量门/防重复 -> 00 正文根 -> 正文动作 -> 0B 变量赋值 -> 00`。如果复读看到 `02 -> 00(body) -> 25/2E/36/3F...`，触发条件已经落进正文块，事件不会被战场循环测试到。
- 后续会用 `4C/55` 召唤的命名单位，必须在初始 `46/47` 中先以隐藏/备用记录预埋，并用 `48` 提前写装备；不要让后续事件第一次引用完全没有战场记录的 Data。三英实测：`4C` 不会凭空生成祁戍，缺 `47` 隐藏记录会导致后续相邻测试、单挑和阶段剧情不触发；`55` 是复活/重置已存在单位，不是预部署替代品。
- `46/47` 出场标志需按样本复读：当前三英与刘备样本共同观察到 `0=初始可见`、`1=隐藏 / 后续由 4C 出现`。初始友军释斐应写 `46 flag=0`；阶段增援祁戍、影侍应写 `47 flag=1`。若未来发现其它基底语义不同，必须以目标基底样本复读为准。
- 坐标生成要同时查地图视觉、Hexzmap 地形和触发命令。守坛小兵若要“站在祭坛上”，`47` 坐标必须等于 `25` 触发坐标；若写到外圈，会出现视觉偏一格。增援 / 复活点不得落在 `0C 小河`、墙、岩山等不可站立格，写前必须复读 terrain。
- `R_00/S_00` 采用脚本编号到地图槽的默认绑定：`S_00` 对应 `M000`，当前 6.5 布局中 `0x44 战斗初始化` 为零参数，不要在 `0x44` 里幻想填写地图号。地图槽必须先按实际 S 编号、底图尺寸和 Hexzmap 块复读锁定，再规划坐标。
- 《三英战龙帝》实测：`M000.JPG=960x960`，对应 `20x20`；Hexzmap `M000` 为 400 格、段长 402。把旧 `M056 37x30` 设计直接套到 `S_00` 会导致坐标越界和事件错位；`M001/S_01` 现为归档副本。
- 战前准备出战位复读口径：`0x4A` 是强制出战/禁用列表，首值为强制数量，后续为人物 Data ID；`0x4B` 是我军候选出战位，参数为 `出战顺序 / X / Y / 方向 / 标志`，第一个参数不是人物号。强制人物按 `4A` 顺序绑定到 `4B` 候选位顺序。
- 旧版完整树写 `0x05 变量测试` 时必须使用两个 VariableArray 参数形态，不能当作普通短整数参数写；第一组是“必须为真”的变量列表，第二组是“必须为假”的变量列表，否则变量判定会被读歪。
- R 场景坐标和 S 战场坐标不是同一套。自动生成 `0x30/0x32` 前必须按 `pixelX = 8 * (x - y + 42)`、`pixelY = 4 * (x + y - 50)` 校验可见范围；对白节奏用 `0x09 延时`，独立首关默认 `3-5`，不要把所有对白连续平铺。需要“角色从边缘入场”时，不能只写目标点 `0x30`；应先确认目标基底绘图顺序。三英独立首关当前锁定为 `1C 绘图 -> 30 边缘出现 -> 09(3) -> 32 移动到目标 -> 09(3)`，再按需要追加 `33 转向` / `34 动作`。武将必须先出现才能移动；`1E 重绘`只作显示异常时的刷新保险，不作为默认硬插命令。中文剧情建议用 UTF-8 蓝图或资源文件维护；每条文本参数可用换行分隔语气，但换行符最多 2 个，再由完整树写入器生成 `.eex`，避免在乱码源码里硬编码台词。
- `0x11 剧本跳转` 的目标参数按本地工具显示和复读解释：`0 -> S_00.eex`、`1 -> R_01.eex`、`2 -> S_01.eex`、`3 -> R_02.eex`。该映射只用于明确采用剧本跳转的路线；当前推荐首关战前准备路线不在 R 侧硬写 `0x11`，而通过玩家切换 Scene 的 `0x0D` 结束 Scene 让引擎进入对应 S。
- 低位 Data 号能让原生撤退台词、特殊暴击类型、部署引用和后续扩展更直观；但低位覆盖会破坏原剧情兼容性，只能用于独立基底。
- 6.5 职业名称槽位长度有限，长中文职业名可能溢出；表内用短名，完整称号放在列传、说明和 R/S 文本中。
- 物品分段必须遵守 6.5 段规则：武器在武器段，防具从 `DefID=70` 段开始，辅助从 `AssID=109` 段开始，不能为了编号整齐把防具或辅助塞进武器段。
- 自定义“个人特技 / 兵种特技”不能伪造不存在的引擎特效。可见名称写入列传、装备说明和 R/S 提示；实战效果优先由 S 事件或已验证装备特效实现。

## 实现/使用方法

### 自动化总流程

| 阶段 | 动作 | 输出 | 失败处理 |
| --- | --- | --- | --- |
| 1. 需求归类 | 判断是普通改表、独立单关、整包 MOD 还是独立精简基底 | 制作模式和目标基底 | 信息不足时先补设计，不写文件 |
| 2. 模板选择 | 独立首关选三英 `R_00/S_00`，后续关参考同模组样本或刘备 `R_01/S_01`，纯文本替换只走文本模板 | R/S 路线和样本文件 | 找不到同类样本则停止到蓝图，不写结构 |
| 3. 知识检索 | 读取本页、命令映射表、剧本制作页、刘备案例页、数据表与资源页 | 可写规则清单 | 找不到证据则列为待验证 |
| 4. 设计定稿 | 输出 `StandaloneScenarioDesign / ModDesign`，锁死 R/S、Data、变量、奖励、地图、资源门控 | 设计稿和槽位表 | 不允许写包时临场换 ID |
| 5. 命令蓝图 | 将 R/S 文案拆成文本宏和完整树命令蓝图，标注命令号、参数槽、变量和风险 | 可导入文本 + 完整树蓝图 | 不能只保留自然语言事件 |
| 6. 只读复读 | 复读人物、职业、物品、R/S、地图、Hexzmap、资源槽 | 写前报告 | 冲突则回改设计稿 |
| 7. 预览包 | 运行 `preview_mod_package` 或对应预览工具 | 预览报告 | 只修复明确错误，不放宽护栏 |
| 8. direct 写入 | 运行 `apply_mod_package(automation_mode=direct)`、`apply_scenario_patch_aggressive` 或专用发布工具 | 写入报告、备份和复读结果 | 禁止绕过专用写入器裸改文件 |
| 9. 写后复读 | 复读表、文本、资源、R/S 命令树、地图地形 | 写后报告 | 不一致则按备份恢复或修包重写 |
| 10. 烟测/实机 | 验证 R 打开、R 跳 S、S 载图、部署、事件、胜败、奖励 | 验证报告 | 失败项回设计或回包修复 |
| 11. 归档 | 导出 `auto_validate_mod` / `export_mod_report` 报告，记录备份路径和剩余风险 | 正式创作记录 | 未验证项必须留在报告中 |

### 独立基底清理与低位编号

独立基底用于“从旧 6.5 基底抽出一个新 MOD 工程”。自动化脚本应复制原基底到新目录，排除 `_CCZModStudio_Backups`、`SV` 和临时输出，再按以下规则处理：

| 类别 | 首关建议 | 扩展预留 | 规则 |
| --- | --- | --- | --- |
| R/S | 正式首关 `R_00 / S_00` | `R_01/S_01` 起顺序扩展，`R_90/S_90` 后留给烟测 | 旧 R/S 可移入 `RS/_DisabledOriginalRS`，不删除；`R_00` 为剧情/出战设置 Scene + 玩家切换 Scene，`S_00` 为正式战场 |
| 人物 Data | `0-8` | `9-31` 本章，`32-63` 敌军精英，`64-127` 后续核心 | 非白名单行写“空/预留”，不缩表 |
| 详细职业/详细兵种 | `0/3/6/9/12/15/18/21/24` | 每系 `+1/+2` 作为二三阶或预留，后续从 `27+` 扩展 | 6.5 常按三行归属一个兵种系；长名用短显示名，完整名写文本 |
| 兵种系 | `0-8` | `9-31` 新兵种系，`32+` 预留或兼容 | 写地形发挥、移动消耗、相克；人物表不直接引用兵种系 |
| 武器 | `0-8` | `9-31` 后续武器 | 只写武器段字段 |
| 防具 | `70-78` | `79-98` 后续防具 | 遵守 `DefID=70` |
| 辅助 | `109-117` | `123+` 按项目规划 | 遵守 `AssID=109` |
| 奖励道具 | `118-122` | 与辅助扩展错开 | 写前必须复读是否空闲 |

低位重占用的关键约束：

1. 原基底仍要保留，不在原基底低位覆盖主角和名将。
2. 独立基底低位覆盖前必须输出写前报告，写后必须复读对应行。
3. 槽位冲突时只允许回到设计稿调整，不允许写包阶段自动换相邻 ID。
4. 低位人物可写原生撤退台词；高位人物若超出原生表覆盖范围，撤退文本必须在 S 事件中兜底。

### Data、职业、物品写回要点

| 数据 | 必填字段 | 复读点 |
| --- | --- | --- |
| 人物 | 名称、职业、等级、五维、HP/MP、头像、R/S 形象、装备、列传、暴击类型、撤退文本 | `Data.e5`、列传、暴击台词、撤退台词、R/S 形象字段 |
| 职业 | 短显示名、移动力、攻击范围、穿透、攻防精爆士、HP/MP 成长、策略列、兵种定位 | `6.5-4`、`6.5-4-2`、`6.5-4-3`、`6.5-5` |
| 武器/防具/辅助 | 名称、类别、图标、初始能力、成长、装备特效号、特效值、说明文本 | 物品表、说明表、图标资源 |
| 奖励道具 | 名称、类别、图标、可用性、数量/效果、说明文本 | 物品分段和 R/S 奖励命令引用 |

资源未就绪时，头像、R 形象、S 形象字段保持 `0`。自动化不能为了“看起来完整”写入未预览的资源编号。若使用现有资源补齐人物形象，按以下顺序执行：

| 步骤 | 要求 |
| --- | --- |
| 1. 映射确认 | R 形象按 `R=n -> Pmapobj.e5 #2n+1/#2n+2`；S 形象按 `S=0` 职业默认、`S=1..32` 特殊三阶、`S>=33` 特殊单图 |
| 2. 候选预览 | 读取 E5 `0x110` 索引表，渲染 `Pmapobj.e5` 和 `Unit_mov/atk/spc.e5` 候选缩略图；不能按裸扫顺序猜编号 |
| 3. 角色匹配 | 为每名角色记录 R/S 编号、资源图号和选择理由，避免只写一个数字 |
| 4. 写入与复读 | 写 `Ekd5.exe` 的 `6.5-0-4 R形象`、`6.5-0-5 S形象` 后，复读人物 0-N 的 R/S 编号和资源映射 |

### 物品差异化与中文编码

物品写回时，不能只改名称。每件关键装备至少要明确 `类型 / 图标 / 初始能力 / 成长 / 装备特效号 / 特效值 / 说明文本`，并在写后复读这些字段。

| 场景 | 规则 |
| --- | --- |
| 武器 / 防具 | `装备特效号` 基本可作为实际装备特效解释；同一关的核心装备应避免图标、特效号、特效值三者完全重复 |
| 辅助 / 道具 | 6.5 中 `装备特效号=2/3` 多数是类别标记；实际显示差异优先看 `类型` 字段，宝物页解释应写入项目侧 `CCZModStudio_Notes/*_ItemEffectCatalog.json` |
| 奖励道具 | 奖励也要拆开图标和类型；不要让不同奖励都保持 `icon=0/effect=0/type=60` 这种不可读状态 |
| 中文写入 | `Data.e5 / Star.e5 / Imsg.e5` 表内中文按 GBK 定长写入；项目侧 JSON 用 UTF-8。写入后必须复读中文，若出现 `???` 说明管道或脚本编码已损坏，必须立刻重写文本 |
| 受控二进制写入 | MCP/表写入器不可用时，允许按已确认 `HexTable.xml` 偏移受控写入，但只限当前明确目标项目，并且必须写前备份、写后复读、输出报告 |

### R/S 结构写回要点

R/S 自动化分两条输出线：

| 输出线 | 内容 | 写入方式 |
| --- | --- | --- |
| 文本导入宏 | `@dialog/@narration/@text/@appear/@move/@turn/@action/@battle-turn/@battle-action` | CCZModStudio 文本导入；只处理已定稿文本和轻量演出 |
| 完整树蓝图 | Scene/Section、`27/1C/24/44/45/46/47/4B/07/06/0D/5A`、触发、胜败、变量和跳转 | `apply_scenario_patch_aggressive` / 旧版完整树写入器；必须备份、复读和实机 |

从自然语言直接生成 `.eex` 是禁用路径。任何战场机制都必须先落到命令号、参数槽、变量、防重复和写回风险。

每个 S 事件都必须具备以下字段：

| 字段 | 要求 |
| --- | --- |
| 事件 ID | 使用稳定编号，如 `E01`、`E22A`，不要只写自然语言段落 |
| 设计语义 | 如“刘生邻接祁戍触发轻量单挑” |
| 触发命令 | 查映射表，例如 `0x2E 武将相邻测试` |
| 动作命令 | 如 `0x14 对话`、`0x50 战场动作`、`0x53 撤退`、`0x38 能力变化` |
| 变量 / 防重复 | 使用真实局部变量号，如 `Var7 scale_broken_liu` |
| 写回风险 | 标注 `V1安全 / 完整树 / 高风险 / 待核验` |

常见语义落点：

- 单挑：轻量实现优先用 `0x2E 武将相邻测试`，再接对话、战场动作、撤退、奖励；完整单挑命令族 `0x5F-0x68` 另行实机验证。
- R 路由：独立首关推荐 `R_00 -> S_00`。`R_00` 为正式剧情与出战入口，不要拆文学分幕为真实 Section，不要塞 S-only 命令。当前实机确认路线是双 Scene：剧情 Scene 末尾写 `4B x15 -> 08 true -> 06 1,3,0,1,2,-1... -> 00`；玩家切换 Scene 写 `07 -> 00 -> 08 false -> 出发对白 -> 06 0,0,-1... -> 1D -> 24 255 -> 0D -> 00`。首关战前准备路线不在 R 侧硬写 `11 raw=0`。
- 奖励：用 `0x3D 获得物品`，必须配变量防重复和已复读物品 ID。
- 坐标触发：用 `0x25 进入指定地点测试` 或已核验参数边界的 `0x26 区域测试`，坐标必须来自地图方案和 Hexzmap 复读；未核验 `0x26` 前，人物靠近事件优先降级为 `0x2E` 邻接测试。
- 阶段推进：用局部变量记录 `phase` 只作文本 / 复读辅助；真实 `0x05` 判定要拆成 true-list / false-list 的布尔门，不写 `phase=max()` 或 `Var1==2` 这类伪表达。
- Boss 护阵：优先用 S 事件实现，例如 phase=1 击退 Boss 后 `0x55` 复活/重置并提示护阵未破；不写无法验证的“伤害无效”抽象机制。
- 胜败：玩家显示目标和实际判定分开写；`0x42/0x43` 判定、`0x49/0x0E/0x0D` 收尾由完整树或 force-open 预览生成并复读。

### 地图和资源门控

地图必须按双层实现：

| 层 | 工具/文件 | 要求 |
| --- | --- | --- |
| 视觉底图 | `replace_map_image`、`Map/Mxxx.JPG` | 尺寸必须匹配地图槽，例如 M056 为 `1776x1440`、`37x30` 格 |
| 地形逻辑 | `write_hexzmap_block`、`Hexzmap.e5` | 必经格、触发格、Boss 八邻格必须可通行；墙、河、建筑只放在非必经区域 |

地图写入顺序：先预览底图尺寸，再用 `publish_map_workbench_bundle` 同步发布 `Map/Mxxx.jpg` 与 `Hexzmap.e5` 地形格，最后复读关键坐标。不要新增没有 Hexzmap 绑定的新地图号。

归档路线 `R_01/S_01` 地图特别规则：若未来另行验证并启用 `R_01/S_01`，必须复读并使用 `M001`。历史案例为 `Map/M001.JPG=960x960`、`20x20`、Hexzmap `M001=400` 格；`0x44` 不提供地图号参数，不能靠 S 命令把 `S_01` 临时切到 `M056`。当前默认首关仍是 `R_00/S_00/M000`；如果确实要使用 `M056` 这类大图，应先确认脚本编号、启动路由或引擎地图映射，而不是只改设计稿坐标。

资源写入顺序：先生成或导入素材，再 `preview_e5_image_replace / preview_e5_image_batch_replace`，确认尺寸、索引和调色板，再 direct 写入。AI 绘图到替换可用 `draw_and_replace_ccz_image_asset` 串联完成。没有素材时，不写形象编号。

## 风险与边界

- 本页总结的是 6.5 未加密基底和《三英战龙帝》实施经验，不自动适用于 6.6/6.6x。
- “清旧信息”不是删除表行或缩短文件，而是保留定长结构并写成空/预留。
- `ModPackage V1` 的安全命令白名单不覆盖所有真实剧本命令；非白名单命令可以写入设计映射，但落盘必须走完整树写入器、备份、复读和实机验证。
- `0x76`、跨 Section 跳转、复杂子事件重排和完整单挑命令族属于高风险，不得手填偏移。
- 自定义特技名只有显示价值时，必须明确“由 R/S 事件实现”或“由已验证装备特效实现”；不能写成未验证引擎能力。
- 当前 MCP 正式工具面不再提供测试副本推广入口；正式创作以 direct 写入当前目标项目为准，同时保留备份、复读、报告、路径边界和格式长度护栏。
