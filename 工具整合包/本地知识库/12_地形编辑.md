# 12 地形编辑

## 已确认文件与当前解释

| 文件 | 当前用途 | 当前实现状态 |
| --- | --- | --- |
| `Hexzmap.e5` | 战场变量尺寸地形格数据 | 已支持按目录表读取地图对应地形块、可视化叠加、画笔修改、保存、备份、结构化报告、复读校验 |
| `Map\Mxxx.jpg` | 战场地图底图 | 地图制作页可读取并与地形层叠加；图片替换/地图制作仍需继续与实际战场引用对齐 |

当前 `HexzmapProbeReader` 已改为由 `Map\Mxxx.jpg` 分辨率决定地形格尺寸：`gridWidth = 图片宽度 / 48`，`gridHeight = 图片高度 / 48`。图片宽高必须能被 48 整除，否则不能安全解释为曹操传战场格。`Hexzmap.e5` 中每个地形格 1 字节，单个地形段的地形数据长度应为 `gridWidth * gridHeight`。

已确认的 `Hexzmap.e5` 结构要点：

- 文件绝对偏移 `0x110` 开始是目录表。
- 每条目录项 12 字节，按大端读取：`segmentLength`、`segmentLengthCopy`、`fileOffset`。
- 地形数据起点为 `fileOffset + 2`。
- 地形数据长度为 `gridWidth * gridHeight`；常见 `segmentLength = gridWidth * gridHeight + 2`。
- 保存时必须使用目录项中的真实 `DataOffset`、`Width`、`Height` 写回，不能再按固定块长推算。

## 当前实现

- `地形编辑` 页可以读取 `Hexzmap.e5`，按地图底图分辨率显示对应的变量尺寸地形网格。
- 支持常用地形预设、地形 ID 画笔、撤销/重做。
- `地图制作` 页可把 `Map\Mxxx.jpg` 底图与 Hexzmap 地形层叠加，勾选地形画笔后可直接修改地形层。
- `HexzmapEditorService.SaveBlock` 保存前自动备份 `Hexzmap.e5`，保存后可复读校验，并生成结构化报告，报告类型为 `Hexzmap地形格写入`。

## MCP 无头地形读取闭环

- 2026-06-04 起，MCP 新增 `list_hexzmap_blocks` 和 `read_hexzmap_block`。
- `list_hexzmap_blocks` 复用 `HexzmapProbeReader`，返回每个 `Mxxx` 地形块的底图名、图片分辨率、格子尺寸、目录偏移、数据偏移、段长度、主地形、高频地形、未知地形 ID 和 `CanEdit`。
- `read_hexzmap_block` 按 `map_id` 读取单块地形格矩阵，返回 `cellCount`、`topTerrains`、按行排列的 `TerrainIds`/`TerrainHex` 和行内高频地形摘要。
- 这两个工具只读，目的是让自动化流程先定位块、核对坐标和地形 ID，再决定是否调用 `write_hexzmap_block`。写入仍受 `ProjectVersionGuardService`、地图格数匹配、备份、复读和结构化报告约束。
- 当前基底 `Hexzmap.e5` 尺寸与护栏基准仍不一致时，MCP 可以读取和解释地形块，但不能绕过护栏强写。

## 2026-05-31 已落地：地图/地形尺寸统一

- 已确认创作者规则：`48x48 像素 = 1 个战场格子`。
- 地图尺寸示例：`960x960 -> 20x20`，`1776x1440 -> 37x30`。
- 旧实现把 Hexzmap 全部按固定 `30x30` 解释是错误方向，尤其不能在不匹配地图上继续强写。
- 当前代码已改为地图制作与地形编辑共用同一套尺寸模型：`Map\Mxxx.jpg` 分辨率决定地形网格宽高、坐标换算、叠加预览和写回块长度。
- 地形编辑页、地图制作页、战场地图预览、剧本地图预览都应使用同一个块尺寸来源，不能再局部写死固定格数。

## 2026-06-01 规则再次固化：地形格数必须由底图分辨率/48 推导

- 统一规则：地图/地形网格格数由同编号底图 `Map\\Mxxx.jpg` 的分辨率决定：
  - `gridWidth = imageWidth / 48`
  - `gridHeight = imageHeight / 48`
- 结论：任何写死 `30x30` 的逻辑都只能作为历史探针，不允许进入读写主流程；真实写回必须以目录项 `DataOffset + Width/Height` 为准。

## 2026-05-31 规则固化：按底图分辨率/48 计算地形格尺寸

- 地形格尺寸不是固定 30x30，而是由同编号底图 `Map\Mxxx.jpg` 的分辨率决定：
  - `gridWidth = imageWidth / 48`
  - `gridHeight = imageHeight / 48`
- 写回边界必须基于真实目录项：
  - 地形数据起点：`DataOffset = fileOffset + 2`
  - 地形数据长度：`gridWidth * gridHeight`
  - 段长度应匹配：`segmentLength == gridWidth*gridHeight + 2`（不匹配则拒绝写回）
- 这个规则用于：地形编辑画布、地图制作画布地形叠加、鼠标坐标换算、网格绘制、Hexzmap 段长度校验与写回长度计算。

## 2026-05-31 结构证据补记：Hexzmap.e5 很可能不是固定连续 900 字节块

- `Hexzmap.e5` 文件长度：`45254`，扣除 `Ls12` 头后载荷长度约 `45238`。
- 当前项目 `Map` 目录共有 `59` 张战场底图。
- 这 `59` 张底图按 `48x48 像素 = 1 格` 换算后的总格数约为 `43178`。
- `Hexzmap.e5` 载荷与地图总格数的差值约为 `2060`，量级上更像：
  - `真实地图格数据`
  - 再加上少量每图元数据/索引/尾部结构
- 在 `payload` 偏移约 `400`、`800` 一带已经出现明显不像普通地形 ID 的 32 位结构数据，因此当前“整文件连续按 900 字节切块”不仅尺寸假设有问题，连块边界本身也很可能是错的。
- 当前项目已做的安全收口：
  - 地图制作页网格改为按底图真实格数显示；
  - 地形编辑页和地图制作页使用同一套 `Width/Height/DataOffset`；
  - 保存时按目录项真实偏移写回 `Width * Height` 字节，并在写后复读校验；
  - 图片宽高不能被 48 整除时，不应启用对应地形层读写。

## 2026-05-31 目录表猜测补记：疑似存在“格数+2”列

- 在 `Hexzmap.e5 payload` 偏移约 `392` 开始，出现大量连续 32 位大端整数三元组。
- 其中反复出现的值包括：
  - `402`
  - `562`
  - `578`
  - `642`
  - `674`
  - `770`
  - `786`
  - `802`
  - `962`
  - `1026`
  - `1282`
  - `1602`
- 这些值与实际地图格数存在明显对应关系：
  - `20x20 = 400`，对应 `402`
  - `28x20 = 560`，对应 `562`
  - `24x24 = 576`，对应 `578`
  - `32x20 = 640`，对应 `642`
  - `32x32 = 1024`，对应 `1026`
  - `40x32 = 1280`，对应 `1282`
  - `40x40 = 1600`，对应 `1602`
- 当前最强候选解释：
  - 目录表里至少有一列记录的是 `地图格总数 + 2`
  - 另外一列可能是累计偏移或数据起点
  - 三元组的完整语义仍需继续核实
- 这已经足以支持后续新建“Hexzmap 目录表探针”，按地图真实格数去反向匹配每张地图的数据段，而不是继续按固定 `900` 字节切块。

## 2026-05-31 目录表核心关系已确认并校正

- 对 `Hexzmap.e5` 目录表做连续验证后，当前确认目录项格式为：
  - `A = 当前段长度`
  - `B = 当前段长度副本`
  - `C = 当前段文件绝对偏移`
- 目录表起点为文件绝对偏移 `0x110`，每条 12 字节，三列均按大端解释。
- 数据段内部前 2 字节不是地形格数据；当前读写地形数据从 `fileOffset + 2` 开始。
- 因此，地形可写长度不是 `segmentLength`，而是 `segmentLength - 2`，并应与地图底图换算出的 `gridWidth * gridHeight` 一致。
- 因此，`Hexzmap.e5` 至少包含一个明确的“目录表 + 数据段”结构，而不是简单的连续固定 `900` 字节块。
- 进一步确认：
  - 最后一条有效段满足 `off + len == 文件长度`
  - 例如：`off=44852 len=402 end=45254`，正好等于 `Hexzmap.e5` 文件长度 `45254`
  - 因此目录偏移不是 `payload` 相对偏移，而是文件绝对偏移
- 目录尾部还存在至少一条明显无效的垃圾记录：
  - `off=3584 len=1010565120`
  - 这条应在读取器中剔除，不能参与地图段解释
- 当前仍未确认的部分：
  - 这 `49` 条有效段如何映射到当前项目 `59` 张地图；
  - 每个数据段前 2 字节的真实含义；
  - `Hm??.e5` 与这些段之间的对应关系。
